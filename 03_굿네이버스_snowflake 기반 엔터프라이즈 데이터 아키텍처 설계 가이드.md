## 1. 아키텍처 핵심 원칙

- **배치 중심의 간결한 수집**: 월/일 단위 리포팅 목적(월 100회 내외 조회)이므로 연속 스트리밍 인프라(Kafka, Snowpipe Streaming)를 배제한다. 단, **파일 기반 배치 landing**(스테이지에 주기적으로 떨어지는 export 파일)은 `COPY` 또는 **Snowpipe**(파일 트리거)로 처리한다. → 배제 대상은 *연속 스트리밍*이지 Snowpipe 자체가 아니다.
- **소비 계층의 완전 분리**: 현업 BI용 통합 플랫 뷰(One-Big-Table)와 Cortex LLM용 시맨틱 뷰(Semantic View)를 엄격히 분리 운영한다.
- **제약 조건의 논리적 방어**: Snowflake는 PK/FK를 물리적으로 강제(enforce)하지 않으므로(정보성), 데이터 정합성 검증 파이프라인을 필수로 구축한다.

## 2. 데이터 모델링 및 소비 계층 (Consumption Layer)

최종 소비 주체(사람 vs AI)에 따라 데이터 제공 형태를 다르게 가져간다.

| 구분 | 소비 주체 | 제공 형태 | 목적·특징 |
|---|---|---|---|
| **현업/BI용** | 사람, 대시보드 | 통합 플랫 뷰 (`WIDE_*`) | 복잡한 JOIN 생략, 직관적 탐색·필터링 |
| **Cortex용** | AI (LLM) | 시맨틱 뷰 (Semantic View) | 스타 스키마 기반 메타데이터 제공, 정확한 SQL 생성 유도 |

### 2.1. 현업/BI 조회용: 통합 플랫 뷰 (Integrated Flat View)

스타 스키마의 조인을 미리 평탄화하여 하나의 넓은 테이블 형태로 제공하는 모델(`WIDE_MEMBER_MONTHLY` 등 9종).

- **목적**: 현업이 SQL 조인 없이 직관적으로 조회·대시보드를 구성.
- **구현 방식**:
  - 초기에는 갱신 비용을 최소화하기 위해 **논리적 뷰(Logical View)** 로 제공(현재 조회량 기준 최적).
  - Snowflake의 **Materialized View는 다중 테이블 조인을 지원하지 않으므로 사용하지 않는다.**
  - 향후 성능 병목 발생 시 **Dynamic Table** 또는 **Task + CTAS**로 물리화를 검토.

### 2.2. Cortex(LLM)용: 시맨틱 뷰 (Semantic View)

통합 플랫 뷰를 LLM에 직접 연결하지 않는다. Cortex Analyst는 뷰의 원문(Row)을 읽어 요약하는 것이 아니라, **메타데이터 기반으로 SQL을 생성(NL-to-SQL)** 하여 쿼리를 수행한다.

- **목적**: LLM 환각 방지, 정확한 집계 쿼리 생성 유도.
- **구현 방식**:
  - 통합 플랫 뷰가 아닌 **코어 스타 스키마 위**에 시맨틱 레이어를 정의한다(cross-fact 분석·conformed dimension 활용).
  - 조인 관계, 데이터 단위(Grain), **비가산 지표**(율·평균), **파생 지표(Derived Metric)** 를 명시적으로 관리하여 잘못된 연산(중복 SUM)을 방지.
  - **비용 관리**: 토큰 비용은 스캔 행 수가 아니라 **스키마·메타데이터 크기**에 좌우되므로, 시맨틱 뷰를 간결·명확하게 정의하는 것이 핵심.

## 3. 리스크 방어 및 정합성 관리 (Data Quality & Governance)

### 3.1. Fan-out 방지 및 PK/FK 검증

- Snowflake는 PK/FK를 문서용(정보성)으로만 활용하며 강제하지 않는다.
- 차원 키가 중복되면 통합 플랫 뷰 생성 시 **조용한 Fan-out(데이터 배수 증식)** 이 발생하여 심각한 지표 오류를 초래한다(에러 없이 틀린 숫자 → 더 위험).
- **방어는 두 층으로 구분**:
  - **`QUALIFY` dedup** → SCD2 차원의 *정당한 다중버전*(설계상 정상) 대응. (`DIM_MEMBER` 등)
  - **DMF(Data Metric Functions) 유일성 검증** → *ETL 버그성 SK 중복* 대응. **15개 DIM 전체 PK**에 적용(SCD2 하나로 한정하지 않는다).
- 데이터 적재 완료 직후 DMF 또는 유일성 검증 쿼리를 **필수 파이프라인**으로 구성한다.

### 3.2. 소비 계층 거버넌스 (RBAC + PII 마스킹)

- 통합 플랫 뷰에는 회원 **성별·지역·연령대·회원번호** 등 개인정보성 컬럼이 노출된다.
- **접근 롤 분리(RBAC)**: 플랫 뷰 조회 권한을 현업 롤로 한정하고, 코어 스타 스키마와 분리.
- **마스킹 정책(Masking Policy)**: PII 컬럼에 태그 기반 마스킹을 적용하여 비인가 롤에는 가림 처리.

### 3.3. SCD Type 2 (이력 관리) 처리

- 변경 이력이 남는 차원(회원·조직)은 `QUALIFY` 구문으로 Dedup하여 **DK당 1행(최신 버전)** 만 조인되도록 방어.
- **회원 속성 = 현재버전-only로 확정**됨(팩트가 `MEMBER_DK`만 보유). As-was(과거 시점) 분석은 팩트에 `MEMBER_SK` 추가(스키마 변경) 후 별도 뷰로 제공 — 현재 범위 아님.
- **조직 = as-was 확정**(팩트가 버전 SK `ORG_SK` 보유 → PK 직결, `IS_CURRENT` 미사용).

## 4. 파이프라인 및 성능 최적화 (Pipeline & Performance)

- **Silver → Gold 변환**:
  - **비-SCD 팩트·플랫 뷰**: 갱신 주기에 맞춘 **Dynamic Table**로 선언형 자동화.
  - **SCD2 차원**(`DIM_MEMBER`·`DIM_ORG`): 이전 상태 비교·버전 마감이 필요하므로 Dynamic Table로 표현 불가 → **Stream + Task + MERGE**로 구성.
- **마이크로 파티션 프루닝**:
  - 프루닝은 **뷰가 아니라 하위 FACT 테이블의 적재순서·클러스터링**에 좌우된다. 플랫 뷰는 predicate를 하위 FACT로 내려보낼 뿐이다.
  - 스캔 바이트·컴퓨트를 줄이는 메커니즘이며 **LLM 토큰 절감과는 무관**하다.
- **클러스터링 키**:
  - 사전 정의하지 않는다. 런칭 후 실제 쿼리 패턴·실측 병목(partition depth 등)을 모니터링한 뒤 필요한 테이블에만 선택 적용.

## 5. 설계 및 구축 우선순위 (Roadmap)

1. **시맨틱 뷰(Semantic View) 설계 (최우선)** — Cortex Analyst 기반 작업. 스타 스키마 위 차원·측정값·파생지표 정의.
2. **데이터 품질 검증(DMF) 파이프라인** — PK/FK 미강제 Fan-out 방어. 적재 후 15개 DIM PK 유일성·정합성 검증.
3. **소비 계층 거버넌스** — 플랫 뷰 RBAC 롤 분리 + PII 마스킹 정책.
4. **Silver → Gold 배치 자동화** — 비-SCD는 Dynamic Table, SCD2는 Task+MERGE.
5. **통합 플랫 뷰 논리 모델링** — 현업용 뷰 제공, 물리화는 보류(조회량 기준).
6. **클러스터링·성능 튜닝** — 운영 단계 실측 기반 사후 적용.

|---|---|
| "빅테이블 = Google Bigtable(NoSQL)로 스트리밍 수집" | 여기서 빅테이블 = One-Big-Table(플랫 뷰). 원천은 배치성 CRM·ERP·GA4 |
| "플랫 뷰를 LLM 컨텍스트에 꽂으면 토큰 절감" | Cortex Analyst는 행을 안 읽음. Semantic View 메타데이터로 SQL 생성 |

| "파티션 프루닝으로 토큰 절감" | 프루닝은 컴퓨트 비용 절감. LLM 토큰과 무관 |
| "Snowpipe는 스트리밍이라 배제" | Snowpipe는 파일 배치 landing 도구. 배제 대상은 연속 스트리밍 |

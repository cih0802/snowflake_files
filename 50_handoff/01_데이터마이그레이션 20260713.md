# 데이터 마이그레이션 작업 문서 (A → B → 로컬 → C)

> 📎 **연계 파일 (스키마 DDL)**: `02_데이터마이그레이션 GN_DW_BRONZE_DDL_20260713.sql`
> `snow://workspace/USER$.PUBLIC."snowflake_files"/versions/head/02_데이터마이그레이션 GN_DW_BRONZE_DDL_20260713.sql`
> A에서 `GET_DDL`로 추출한 GN_DW 브론즈 계층 DDL 스냅샷. 3.2(추출)·5.1(재생성) 단계에서 사용.

## 1. 개요

본 문서는 아래 3단계 경로로 데이터를 이관하는 작업 절차를 정의한다.

```
[A 계정]  --(Direct Share, 동일 리전)-->  [B 계정]
[B 계정]  --(로컬 물리적 다운로드, CSV+GZIP)-->  [작업 서버/로컬]
[로컬]    --(업로드)-->                     [C 계정]
```

- **A 계정**: 원본 데이터 보유(Provider)
- **B 계정**: 공유 수신 및 파일 추출/다운로드 수행(Consumer)
- **C 계정**: 최종 적재 대상(Target)

> 전제: A와 B는 **동일 클라우드/동일 리전**이므로 Direct Share가 가능하다.
> C는 리전/클라우드가 다를 수 있으므로 파일 기반(물리적 이관)으로 처리한다.

### 1.1 이번 개정 핵심 (Snowflake 기술문서 기준)
- **파일 포맷: CSV + `COMPRESSION = GZIP`** 로 다운로드한다.
  - `GZIP` 압축은 **CSV/JSON** 언로드에서 지원된다. **PARQUET 언로드는 `SNAPPY/LZO/NONE`만 지원(GZIP 불가)**.
- **전체 스키마 DDL을 먼저 추출**하여 C에 동일 구조로 테이블을 생성한 후 적재한다.
  - CSV 적재는 **위치(순서) 기반**이며 `MATCH_BY_COLUMN_NAME`을 **지원하지 않는다**(해당 옵션은 Parquet/ORC/Avro/JSON 전용). 따라서 **원본과 동일한 컬럼 순서·데이터 타입**이 보장되어야 정확히 적재된다.

## 3. 1단계: A → B (Direct Share, 동일 리전)

### 3.1 A 계정 — Share 생성 및 부여
```sql
-- A 계정 (Provider), ACCOUNTADMIN 역할
USE ROLE ACCOUNTADMIN;

CREATE SHARE mig_share COMMENT = 'A->B 데이터 마이그레이션 공유';

-- 공유할 DB/스키마/테이블에 대한 사용 권한 부여
GRANT USAGE   ON DATABASE  src_db            TO SHARE mig_share;
GRANT USAGE   ON SCHEMA    src_db.src_schema TO SHARE mig_share;
GRANT SELECT  ON ALL TABLES IN SCHEMA src_db.src_schema TO SHARE mig_share;
-- 특정 테이블만: GRANT SELECT ON TABLE src_db.src_schema.tbl TO SHARE mig_share;
-- 주의: ALL TABLES는 '현재 존재하는' 테이블만 대상. 이후 추가된 테이블은 재부여 필요.

-- B 계정을 공유 대상으로 추가 (B의 account locator 또는 org.account 사용)
ALTER SHARE mig_share ADD ACCOUNTS = <B_ACCOUNT_IDENTIFIER>;

-- 확인
SHOW GRANTS TO SHARE mig_share;
SHOW SHARES LIKE 'MIG_SHARE';
```

### 3.2 A 계정 — 전체 스키마 DDL 추출 (적재 정확도 확보)
CSV 적재는 위치 기반이므로 **C에 원본과 동일한 테이블 구조**를 먼저 만들어야 한다.
소유 계정인 A에서 `GET_DDL`로 스키마 전체 DDL을 추출한다.

> 📎 **연계 파일**: 본 단계에서 추출한 브론즈 계층 DDL 스냅샷은 아래 파일에 정리되어 있다.
> `snow://workspace/USER$.PUBLIC."snowflake_files"/versions/head/02_데이터마이그레이션 GN_DW_BRONZE_DDL_20260713.sql`
> (브론즈 4개 스키마: CRM 43 · AGENCY 3 · ERP 1 · GA4 1 = 테이블 48개, 파일포맷 2개, 객체 인덱스 포함)
> GA4 `events_20260501` 은 VARIANT 다수 → 4.4/5.5 Parquet 경로로 이관.

```sql
-- A 계정
-- 스키마 내 지원 객체(테이블/뷰/시퀀스/파일 포맷 등) 전체 DDL을 완전수식명으로 반환
SELECT GET_DDL('SCHEMA', 'src_db.src_schema', TRUE);

-- 개별 테이블만 필요한 경우
SELECT GET_DDL('TABLE', 'src_db.src_schema.tbl', TRUE);
```

- 결과를 `schema_ddl.sql` 로 저장한다(워크시트 결과 복사 또는 아래처럼 스테이지 언로드 후 GET).
- **DDL 파일도 CSV 데이터와 함께 로컬로 다운로드**하여 C로 이관한다.
- C에 그대로 실행하기 전, DB/스키마명을 C 환경에 맞게 치환한다(예: `src_db` → `SANDBOX`).

> 참고: 공유받은(imported) DB에서는 `GET_DDL`이 제한될 수 있으므로 **DDL 추출은 원본 A에서 수행**을 권장한다.

### 3.3 B 계정 — Share로부터 DB 생성
```sql
-- B 계정 (Consumer), ACCOUNTADMIN 역할
USE ROLE ACCOUNTADMIN;

-- 수신 가능한 공유 확인
SHOW SHARES;   -- inbound 항목에서 A의 mig_share 확인

-- 공유로부터 읽기 전용 DB 생성
CREATE DATABASE shared_from_a FROM SHARE <A_ACCOUNT_IDENTIFIER>.mig_share;

-- 접근 권한 부여(작업 역할에)
GRANT IMPORTED PRIVILEGES ON DATABASE shared_from_a TO ROLE mig_role;

-- 데이터 확인
SELECT COUNT(*) FROM shared_from_a.src_schema.tbl;
```

> **주의**: 공유받은 DB는 **읽기 전용**이다. B에서 파일로 추출하려면 스테이지로 UNLOAD해야 한다(아래 4단계).

## 4. 2단계: B → 로컬 물리적 다운로드 (CSV + GZIP)

공유 DB는 직접 다운로드할 수 없으므로, B의 **내부 스테이지로 CSV(GZIP) 언로드(COPY INTO)** 후 `GET`으로 로컬 다운로드한다.

### 4.1 파일 포맷(CSV+GZIP) 및 스테이지 준비
```sql
-- B 계정, mig_role
USE ROLE mig_role;
USE SCHEMA b_work_db.stage_schema;   -- 언로드 결과를 담을 자체 스키마

-- CSV + GZIP 파일 포맷 (적재 정확도 확보용 옵션 명시)
CREATE OR REPLACE FILE FORMAT ff_csv_gzip
  TYPE = CSV
  COMPRESSION = GZIP
  FIELD_DELIMITER = ','
  RECORD_DELIMITER = '\n'
  FIELD_OPTIONALLY_ENCLOSED_BY = '"'   -- 콤마/따옴표/개행 포함 값 보호
  NULL_IF = ('\\N')                    -- NULL 표기 규칙(언로드/적재 동일하게)
  EMPTY_FIELD_AS_NULL = FALSE          -- 빈문자열과 NULL 구분 유지
  TIMESTAMP_FORMAT = 'YYYY-MM-DD HH24:MI:SS.FF9 TZH:TZM'
  DATE_FORMAT = 'YYYY-MM-DD'
  ESCAPE_UNENCLOSED_FIELD = NONE;

CREATE STAGE IF NOT EXISTS mig_unload_stage FILE_FORMAT = ff_csv_gzip;
```

### 4.2 테이블 → 스테이지 언로드
```sql
-- 테이블별로 반복. GZIP 압축 CSV(.csv.gz)로 생성됨
COPY INTO @mig_unload_stage/tbl/
FROM shared_from_a.src_schema.tbl
FILE_FORMAT = (FORMAT_NAME = ff_csv_gzip)
HEADER = TRUE                 -- 컬럼 헤더 포함(적재 시 SKIP_HEADER=1과 짝)
MAX_FILE_SIZE = 268435456     -- 파일당 약 256MB로 분할(기본 16MB, 최대 5GB)
OVERWRITE = TRUE;

-- 단일 파일이 필요하고 5GB 미만이면: SINGLE = TRUE 사용 가능
-- 생성된 파일 확인
LIST @mig_unload_stage/tbl/;
```

### 4.3 로컬로 다운로드 (Snowflake CLI `snow`)
`GET`은 클라이언트(Snowflake CLI `snow` 등)에서 실행해야 한다. Snowsight 워크시트에서는 불가.
```bash
# Snowflake CLI 예시 (B 계정 연결) - 데이터 다운로드(GET)
snow stage copy @b_work_db.stage_schema.mig_unload_stage/tbl/ /data/migration/tbl/ \
  -c <B_CONNECTION> --recursive

# DDL 스크립트(3.2에서 A에서 추출)도 함께 준비 -> /data/migration/ddl/schema_ddl.sql
```

- 다운로드 파일은 `.csv.gz` 형태로 **압축 상태 그대로** 저장된다.
- 다운로드 위치: `/data/migration/<테이블명>/` (용량 확인 필수)
- 다운로드 후 파일 개수 / 총 용량을 `LIST` 결과와 대조

> 대용량인 경우 로컬 대신 **외부 스테이지(S3/GCS/Azure)**를 경유하면 GET/PUT 없이 계정 간 직접 이관이 가능하다(선택지, 6.2 참고).

### 4.4 반정형/바이너리 컬럼 테이블: Parquet(SNAPPY)로 분리 처리
`VARIANT/OBJECT/ARRAY/GEOGRAPHY/GEOMETRY/BINARY` 컬럼은 CSV 왕복 시 손실·오변환 위험이 있다.
**해당 컬럼을 가진 테이블만** CSV 경로에서 제외하고 Parquet로 별도 이관한다.

#### 4.4.1 대상 테이블 탐지 (B의 공유 DB에서 실행)
```sql
-- 반정형/바이너리 컬럼을 포함한 테이블 목록
SELECT table_name, LISTAGG(column_name || ':' || data_type, ', ') AS risky_columns
FROM shared_from_a.information_schema.columns
WHERE table_schema = 'SRC_SCHEMA'
  AND data_type IN ('VARIANT','OBJECT','ARRAY','BINARY','GEOGRAPHY','GEOMETRY')
GROUP BY table_name
ORDER BY table_name;
```
- 위 목록에 **나오는 테이블 → Parquet 경로(4.4.2)**, 나머지 → CSV+GZIP 경로(4.1~4.3).

#### 4.4.2 Parquet 언로드 (SNAPPY)
```sql
-- B 계정. Parquet은 GZIP 미지원 → SNAPPY 사용(반정형/타입 보존에 유리)
CREATE OR REPLACE FILE FORMAT ff_parquet
  TYPE = PARQUET
  COMPRESSION = SNAPPY;

CREATE STAGE IF NOT EXISTS mig_unload_stage_pq FILE_FORMAT = ff_parquet;

COPY INTO @mig_unload_stage_pq/tbl_semi/
FROM shared_from_a.src_schema.tbl_semi
FILE_FORMAT = (FORMAT_NAME = ff_parquet)
HEADER = TRUE                 -- Parquet 컬럼명 기록(적재 시 컬럼명 매칭에 사용)
MAX_FILE_SIZE = 268435456
OVERWRITE = TRUE;

LIST @mig_unload_stage_pq/tbl_semi/;
```
- 다운로드는 CSV와 동일하게 `GET`으로 로컬에 저장(`.parquet`, 이미 SNAPPY 압축이므로 `AUTO_COMPRESS=FALSE`).

## 5. 3단계: 로컬 → C 계정 적재

### 5.1 C 계정 — DDL로 스키마/테이블 재생성 (정확도 핵심)
A에서 추출한 DDL을 **먼저 실행**하여 원본과 동일한 구조를 만든다.

> 📎 **연계 파일**: `02_데이터마이그레이션 GN_DW_BRONZE_DDL_20260713.sql` 를 사용한다.
> `snow://workspace/USER$.PUBLIC."snowflake_files"/versions/head/02_데이터마이그레이션 GN_DW_BRONZE_DDL_20260713.sql`
> 파일 상단의 사용법(USAGE)에 따라 DB/스키마명을 C 환경에 맞게 치환 후 [SCHEMA]→[TABLE] 순으로 실행.
```sql
-- C 계정, 적재 역할
USE ROLE load_role;
CREATE DATABASE IF NOT EXISTS SANDBOX;
CREATE SCHEMA   IF NOT EXISTS SANDBOX.TOOLS;
USE SCHEMA SANDBOX.TOOLS;

-- schema_ddl.sql 내용을 실행 (DB/스키마명은 C 환경에 맞게 치환)
-- 예) Snowflake CLI 파일 실행:
--   snow sql -c <C_CONNECTION> --role LOAD_ROLE -f /data/migration/ddl/schema_ddl.sql
```

### 5.2 C 계정 — 적재용 파일 포맷/스테이지 준비
> ⚠️ 실제 이관에서는 NULL 토큰이 `\\N`(3글자)이어서 아래 `NULL_IF`만으로는 실패했다. 검증된 포맷은 **부록 A.2** 참조.
```sql
-- 언로드 때와 동일한 CSV 규칙으로 적재 포맷 구성
CREATE OR REPLACE FILE FORMAT ff_csv_gzip
  TYPE = CSV
  COMPRESSION = GZIP
  FIELD_DELIMITER = ','
  RECORD_DELIMITER = '\n'
  FIELD_OPTIONALLY_ENCLOSED_BY = '"'
  NULL_IF = ('\\N')
  EMPTY_FIELD_AS_NULL = FALSE
  SKIP_HEADER = 1              -- 언로드 시 HEADER=TRUE 였으므로 헤더 1줄 스킵
  TIMESTAMP_FORMAT = 'YYYY-MM-DD HH24:MI:SS.FF9 TZH:TZM'
  DATE_FORMAT = 'YYYY-MM-DD'
  ESCAPE_UNENCLOSED_FIELD = NONE;

CREATE STAGE IF NOT EXISTS mig_load_stage FILE_FORMAT = ff_csv_gzip;
```

### 5.3 로컬 파일 → C 스테이지 업로드
다운로드 폴더 구조가 `downloaded_data/<SCHEMA>/<table>/파일` 형태라고 가정한다.
```
downloaded_data/BRONZE_CRM/<table>/*.csv.gz
downloaded_data/BRONZE_AGENCY/<table>/*.csv.gz
downloaded_data/BRONZE_ERP/<table>/*.csv.gz
downloaded_data/BRONZE_GA4/<table>/*.parquet     # GA4는 Parquet(SNAPPY)
```
`--recursive` 로 하위 `<table>` 디렉터리 구조를 보존하며 스키마 폴더를 **순서대로** 업로드한다.
CSV 스키마(CRM/AGENCY/ERP)는 CSV 적재 스테이지로, GA4는 Parquet 적재 스테이지(5.5)로 분리한다.

#### Windows PowerShell
PowerShell에서는 줄 연속 문자로 `` ` ``(백틱)을, 배열은 `@( )` 를 사용한다.
```powershell
# 이미 압축 상태이므로 재압축 금지(--no-auto-compress)

# 1) CSV 경로 스키마 (BRONZE_CRM → BRONZE_AGENCY → BRONZE_ERP 순)
foreach ($SCHEMA in @('BRONZE_CRM', 'BRONZE_AGENCY', 'BRONZE_ERP')) {
    Write-Host "=== CSV 업로드: $SCHEMA ==="
    snow stage copy "downloaded_data/$SCHEMA/" `
        "@SANDBOX.TOOLS.mig_load_stage/$SCHEMA/" `
        -c <C_CONNECTION> --recursive --no-auto-compress --overwrite
}

# 2) 반정형(Parquet) 스키마 (BRONZE_GA4) → Parquet 적재 스테이지 (5.5 참조)
snow stage copy "downloaded_data/BRONZE_GA4/" `
    "@SANDBOX.TOOLS.mig_load_stage_pq/BRONZE_GA4/" `
    -c <C_CONNECTION> --recursive --no-auto-compress --overwrite

# 업로드 확인
snow stage list-files @SANDBOX.TOOLS.mig_load_stage/    -c <C_CONNECTION>
snow stage list-files @SANDBOX.TOOLS.mig_load_stage_pq/ -c <C_CONNECTION>
```

#### macOS / Linux (bash)
```bash
for SCHEMA in BRONZE_CRM BRONZE_AGENCY BRONZE_ERP; do
  echo "=== CSV 업로드: ${SCHEMA} ==="
  snow stage copy "downloaded_data/${SCHEMA}/" \
    "@SANDBOX.TOOLS.mig_load_stage/${SCHEMA}/" \
    -c <C_CONNECTION> --recursive --no-auto-compress --overwrite
done

snow stage copy "downloaded_data/BRONZE_GA4/" \
  "@SANDBOX.TOOLS.mig_load_stage_pq/BRONZE_GA4/" \
  -c <C_CONNECTION> --recursive --no-auto-compress --overwrite

snow stage list-files @SANDBOX.TOOLS.mig_load_stage/    -c <C_CONNECTION>
snow stage list-files @SANDBOX.TOOLS.mig_load_stage_pq/ -c <C_CONNECTION>
```

> 이후 5.4(CSV)·5.5(Parquet) `COPY INTO ... FROM` 경로는 스키마 하위 폴더를 반영해
> `@mig_load_stage/<SCHEMA>/<table>/` 형태로 지정한다.

### 5.4 스테이지 → 테이블 적재
```sql
-- C 계정. CSV는 위치 기반 적재이므로 DDL로 만든 컬럼 순서와 정확히 일치해야 함
COPY INTO SANDBOX.TOOLS.tbl
FROM @mig_load_stage/tbl/
FILE_FORMAT = (FORMAT_NAME = ff_csv_gzip)
ON_ERROR = ABORT_STATEMENT      -- 오류 시 전체 중단(정확도 우선). 검토용은 CONTINUE
PURGE = FALSE;                  -- 적재 후 스테이지 파일 유지(검증 후 수동 정리)

-- 적재 결과/오류 확인
SELECT COUNT(*) FROM SANDBOX.TOOLS.tbl;
-- 최근 적재 이력/거부 행 확인
SELECT * FROM TABLE(VALIDATE(SANDBOX.TOOLS.tbl, JOB_ID => '_last'));
```

> `MATCH_BY_COLUMN_NAME`은 CSV에서 사용 불가하다. 반드시 DDL로 **동일 컬럼 순서**를 맞춘 뒤 적재한다.

### 5.5 Parquet 테이블 적재 (반정형/바이너리 대상)
4.4에서 Parquet로 처리한 테이블은 컬럼명 기반 매칭으로 적재한다(CSV와 달리 지원됨).
```sql
-- C 계정. 테이블은 5.1의 DDL로 이미 생성되어 있어야 함
CREATE OR REPLACE FILE FORMAT ff_parquet TYPE = PARQUET COMPRESSION = SNAPPY;
CREATE STAGE IF NOT EXISTS mig_load_stage_pq FILE_FORMAT = ff_parquet;

-- PUT: 이미 SNAPPY 압축이므로 재압축 금지 (Snowflake CLI)
--   snow stage copy "/data/migration/tbl_semi/*.parquet" \
--     @SANDBOX.TOOLS.mig_load_stage_pq/tbl_semi/ \
--     -c <C_CONNECTION> --no-auto-compress --overwrite

COPY INTO SANDBOX.TOOLS.tbl_semi
FROM @mig_load_stage_pq/tbl_semi/
FILE_FORMAT = (FORMAT_NAME = ff_parquet)
MATCH_BY_COLUMN_NAME = CASE_INSENSITIVE   -- Parquet은 컬럼명 매칭 지원
ON_ERROR = ABORT_STATEMENT;

SELECT COUNT(*) FROM SANDBOX.TOOLS.tbl_semi;
```

## 8. 주의사항 / 리스크

- **읽기 전용 공유**: B는 공유 DB를 수정할 수 없으므로 반드시 스테이지 언로드가 필요.
- **CSV+GZIP 적재 정확도**: CSV는 위치 기반 적재 → **DDL로 컬럼 순서/타입을 원본과 동일하게** 재현해야 정확. `HEADER=TRUE` ↔ `SKIP_HEADER=1`, `FIELD_OPTIONALLY_ENCLOSED_BY`, `NULL_IF`, `EMPTY_FIELD_AS_NULL`, `TIMESTAMP_FORMAT` 를 언로드/적재에서 **동일하게** 맞출 것.
- **반정형/BINARY 타입 손실 위험**: `VARIANT/OBJECT/ARRAY/GEOGRAPHY/BINARY` 컬럼은 CSV 왕복 시 손실·오변환 위험. 해당 컬럼이 있으면 그 테이블만 **Parquet(SNAPPY)** 로 별도 처리 권장.
- **압축 재적용 금지**: 이미 `.csv.gz`(또는 `.parquet` SNAPPY)이므로 업로드 시 `snow stage copy --no-auto-compress`.
- **파일 전송은 클라이언트 실행**: Snowsight 워크시트에서는 불가, Snowflake CLI(`snow stage copy`) 필요.
- **파일 크기 제한**: 언로드 파일 최대 5GB, `MAX_FILE_SIZE` 기본 16MB. 대용량은 분할 언로드.
- **용량 산정**: 로컬 다운로드 시 디스크 여유 공간 사전 확인.
- **보안/규정**: 물리적 다운로드 데이터에 PII 포함 여부 확인, 전송/저장 구간 암호화 및 접근 통제.
- **리전 전송 비용**: 계정/리전 간 이관 시 데이터 전송 비용 발생 가능.
- **일관성**: 이관 중 A 원본이 변경되면 스냅샷 시점 기준으로 고정 권장.

### A.2 적재용 파일 포맷 생성 — ⚠️ NULL 토큰 주의
언로드 데이터의 NULL은 **`\\N`(백슬래시 2개 + N, 3글자)** 로 기록돼 있었다(언로드 시 백슬래시가 이스케이프되어 이중화). 숫자 컬럼에서 `Numeric value '\N' is not recognized` 오류의 원인이므로 `NULL_IF`에 반드시 3글자 토큰을 포함한다.
```sql
CREATE OR REPLACE FILE FORMAT SANDBOX.TOOLS.FF_CSV_LOAD
  TYPE = CSV
  COMPRESSION = GZIP
  FIELD_DELIMITER = ','
  RECORD_DELIMITER = '\n'
  FIELD_OPTIONALLY_ENCLOSED_BY = '"'
  SKIP_HEADER = 1                       -- 언로드 시 HEADER=TRUE 였음
  NULL_IF = ('\\\\N', '\\N', '')        -- '\\\\N' → 실제 \\N(3글자), '\\N' → \N(2글자), '' → 빈값
  EMPTY_FIELD_AS_NULL = TRUE
  TRIM_SPACE = FALSE
  ESCAPE_UNENCLOSED_FIELD = NONE;
```
> 확인법: `SELECT $n, LENGTH($n) FROM @stage/... (FILE_FORMAT=>...) WHERE TRY_TO_DOUBLE($n) IS NULL` 로 NULL 토큰의 실제 길이(3)를 확인.

### A.3 스키마 일괄 적재 프로시저
스키마 내 전 테이블을 순회하며 `@stage/<SCHEMA>/<TABLE>/` 에서 COPY 한다.
`FOR rec IN (...) DO ... rec.col`(레코드 점 표기)이 이 환경에서 `invalid identifier` 로 실패하여, **커서 + `FETCH INTO`** 방식으로 작성했다.
```sql
CREATE OR REPLACE PROCEDURE SANDBOX.TOOLS.LOAD_BRONZE_SCHEMA(SCH STRING)
RETURNS STRING
LANGUAGE SQL
AS
$$
DECLARE
  c1 CURSOR FOR
    SELECT table_name FROM GN_DW.INFORMATION_SCHEMA.TABLES
    WHERE table_schema = ? AND table_type = 'BASE TABLE';
  v_tbl STRING DEFAULT NULL;
  v_sql STRING;
  v_cnt INT DEFAULT 0;
BEGIN
  OPEN c1 USING (SCH);
  LOOP
    v_tbl := NULL;
    FETCH c1 INTO v_tbl;
    IF (v_tbl IS NULL) THEN
      BREAK;
    END IF;
    v_sql := 'COPY INTO GN_DW."' || :SCH || '"."' || :v_tbl || '" '
          || 'FROM @SANDBOX.TOOLS.mig_load_stage/' || :SCH || '/' || :v_tbl || '/ '
          || 'FILE_FORMAT = (FORMAT_NAME = SANDBOX.TOOLS.FF_CSV_LOAD) '
          || 'ON_ERROR = ABORT_STATEMENT PURGE = FALSE';
    EXECUTE IMMEDIATE :v_sql;
    v_cnt := v_cnt + 1;
  END LOOP;
  CLOSE c1;
  RETURN 'schema ' || :SCH || ' loaded tables: ' || v_cnt;
END;
$$;
```

### A.7 재적재/멱등성 메모
- `COPY INTO` 는 로드 메타데이터로 이미 적재된 파일을 건너뛴다 → 프로시저 재실행 시 신규 파일만 적재(안전). 강제 재적재는 `FORCE=TRUE`.
- `ON_ERROR=ABORT_STATEMENT` 라 한 테이블 오류 시 해당 COPY 전체 롤백 → 부분 적재 없음.

### A.8 이번에 겪은 함정 (핸드오버 핵심)
1. **NULL 토큰 `\\N`(3글자)**: 기본 `NULL_IF`로는 숫자 컬럼 로드 실패. → A.2의 `NULL_IF` 필수.
2. **스크립팅 레코드 점 표기 실패**: `FOR rec IN (...) DO rec.col` → `invalid identifier`. → 커서 `FETCH INTO` 사용(A.3).
3. **익명 블록 세미콜론 분할**: 툴/CLI가 `;`에서 문을 잘라 파싱 오류 → `EXECUTE IMMEDIATE $$ ... $$;` 로 감싸거나 저장 프로시저로 실행.
4. **GA4 VARIANT**: CSV 경유 시 JSON이 문자열로 저장 → `TRY_PARSE_JSON` 변환 필수(A.5).
5. **대상 테이블 미생성**: DDL 미실행 상태에서 로드하면 "loaded tables: 0". → A.1 선행 확인.

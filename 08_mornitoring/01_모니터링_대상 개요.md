# 01. 모니터링 대상 개요

Snowflake에서 모니터링할 수 있는 대상과 그 수단을 정리한다.
웨어하우스는 가장 기본적인 대상일 뿐, 실제 운영에서는 아래 대상들을 함께 모니터링한다.

## 1. 모니터링 대상

### 컴퓨팅
- **Warehouse** — 크레딧 사용량, 쿼리 큐잉, 로드
- **Query** — 실행 시간, 스필링(spilling), 파티션 프루닝, 실패 쿼리
- **Task / Task Graph** — 실행 이력, 성공·실패, 스케줄 준수
- **Serverless 기능** — Snowpipe, Auto-clustering, Search Optimization, Materialized View, Serverless Task 등

### 데이터
- **Storage** — 테이블/스테이지 용량, Time Travel·Fail-safe 비용
- **Data Quality (DMF)** — Data Metric Function으로 NULL 비율, 중복, 범위 이탈 등
- **Stream** — CDC 상태, 스트림 stale 여부
- **Dynamic Table** — 리프레시 상태, lag
- **Replication** — 복제 지연, 리프레시 이력

### 보안 / 거버넌스
- **Login / Session** — 로그인 실패, 비정상 접속
- **Access History** — 데이터 접근 주체·객체 추적
- **Trust Center** — 보안 스캐너, CIS 벤치마크 준수

### 비용
- **Credit Usage** — 웨어하우스 및 서버리스 크레딧
- **Data Transfer** — 리전 간·클라우드 간 전송
- **Cortex AI** — AI 함수, Agent, Analyst, Intelligence, Cortex Code (자세한 내용은 `02` 문서)

## 2. 모니터링 수단

| 수단 | 설명 | 지연(latency) |
|------|------|---------------|
| `SNOWFLAKE.ACCOUNT_USAGE` 뷰 | 계정 단위 과거 이력 (최대 365일) | 뷰마다 상이 (수 분 ~ 수 시간) |
| `SNOWFLAKE.ORGANIZATION_USAGE` 뷰 | 조직 단위 (Enterprise+, 조직 계정) 사용량 | 최대 24시간 |
| `INFORMATION_SCHEMA` 테이블 함수 | 실시간에 가까운 메타데이터 (짧은 보존기간) | 거의 실시간 |
| Resource Monitor | **웨어하우스 크레딧 한도** 설정·차단 | - |
| Budget | 리소스 그룹 크레딧 지출 **알림** | - |
| Per-user quota (Preview) | 사용자별 크레딧 상한·차단 | - |
| Alert | 조건 충족 시 알림/액션 트리거 | 스케줄 기반 |
| Trust Center | 보안 posture 자동 스캔 | - |

## 3. 중요한 구분: Resource Monitor vs Budget

| 구분 | Resource Monitor | Budget |
|------|------------------|--------|
| 대상 | **웨어하우스 전용** | 계정 전체 또는 지정한 리소스 그룹(서버리스 포함) |
| 동작 | 임계치 도달 시 **알림 + 웨어하우스 suspend(차단 가능)** | 임계치 도달 시 **알림만** (자동 차단 없음) |
| 서버리스/Cortex 적용 | ❌ 불가 | ⭕ 모니터링 가능 (차단은 불가) |

> **핵심**: Cortex AI 같은 서버리스 소비는 Resource Monitor로 제어할 수 없다.
> 알림은 Budget, 하드 차단은 Per-user quota 또는 RBAC로 접근한다. (자세한 내용은 `03` 문서)

# 03. 비용 제한: 예산(Budget) · 사용자별 쿼터 · RBAC

Cortex AI 비용을 통제하는 세 가지 계층을 정리한다.
서버리스 특성상 **Resource Monitor는 사용할 수 없다**는 점이 출발점이다.

## 계층 요약

| 계층 | 수단 | 성격 | 차단 여부 |
|------|------|------|-----------|
| 1. 알림 | Budget | 소프트 | ❌ (알림만) |
| 2. 사용자별 상한 | Per-user quota (Preview) | 하드 | ⭕ (초과 시 차단) |
| 3. 접근 통제 | RBAC (역할/권한) | 하드 | ⭕ (사용 자체 차단) |

---

## 1. Budget — 지출 알림 (소프트 제어)

Budget은 지정한 리소스 그룹의 크레딧 지출을 추적하고 **월 한도 대비 실제/예측 지출이 임계치에 도달하면 알림**을 보낸다.
서버리스 소비도 모니터링 대상에 포함할 수 있다.

- **계정 레벨 Budget**: 계정 전체 크레딧을 추적하는 기본 Budget이 이미 존재하며, 활성화(activate)하면 사용한다.
- **커스텀 Budget**: 특정 객체(웨어하우스, 데이터베이스, 스키마, 테이블, 스테이지 등)를 그룹으로 묶어 추적하는 첫 번째 클래스 객체.

> **한계**: Budget은 **자동 차단을 하지 않는다.** 임계치 도달 시 알림만 전송하며,
> 실제 지출 중단은 운영자의 후속 조치(사용자 차단, 권한 회수 등)에 달려 있다.
> 또한 Cortex 같은 순수 서버리스 소비를 "커스텀 Budget의 리소스"로 개별 지정하는 범위는
> Edition/버전에 따라 다를 수 있으므로, 계정 레벨 Budget으로 전체 AI 지출을 모니터링하는 방식이 가장 안전하다.

### 참고 구문 (실행 전 공식 문서로 재확인 권장)

```sql
-- 커스텀 Budget 생성 (Budget은 SNOWFLAKE.CORE.BUDGET 인스턴스로 관리됨)
CREATE OR REPLACE SNOWFLAKE.CORE.BUDGET my_budget();

-- 월 지출 한도(크레딧) 설정
CALL my_budget!SET_SPENDING_LIMIT(500);

-- 알림 수신 이메일 설정 (사전에 notification integration 및 이메일 검증 필요)
CALL my_budget!SET_EMAIL_NOTIFICATIONS(
    '["admin@company.com"]'
);
```

> `CREATE BUDGET`, 메서드명, 인자 형식은 버전에 따라 차이가 있을 수 있다.
> Snowsight의 **Admin » Cost Management » Budgets** UI를 사용하면 오류 없이 설정하는 것을 권장한다.

## 2. Per-user quota — 사용자별 크레딧 상한 (하드 차단) · Preview

개별 사용자가 짧은 시간에 큰 비용을 발생시키는 것을 방지하기 위한 기능.
대규모 데이터셋에 복잡한 AI 함수를 실행하는 경우 특히 유용하다.

- 월별/일별 **사용자별 크레딧 한도**를 정의한다.
- 계정 전체 사용자 또는 태그로 지정한 특정 사용자(팀/부서)에 적용한다.
- 한도 도달 시 **사용자를 자동 차단**하고, 사이클 리셋 또는 한도 상향 시 자동 해제한다.
- 실제/예측 지출 기준 알림, 임계치별 저장 프로시저 실행(커스텀 액션)을 지원한다.
- 월 사이클은 UTC 달력 월 기준.

> **중요**: Per-user quota는 **Preview 기능**이다. 계정에서 사용 가능한지 확인 후 도입해야 하며,
> 구체적 DDL(객체 생성/설정 구문)은 Snowflake 공식 문서
> (`user-guide/budgets/per-user-quotas`)의 최신 버전을 반드시 참조할 것.
> Snowsight UI에서 사용자별 지출·소진율·차단 상태를 모니터링할 수 있다.

> **정정 사항**: `ALTER USER ... SET RESOURCE_MONITOR_CREDIT_QUOTA` 같은 구문은 **존재하지 않는다.**
> 사용자별 크레딧 상한은 위 Per-user quota 기능으로만 구현한다.

## 3. RBAC — 접근 자체를 제한 (가장 확실한 하드 제어)

비용 발생 이전에 **누가 Cortex를 쓸 수 있는지**를 통제하는 방법이다.

```sql
-- Cortex LLM 함수 사용 권한을 담은 전용 역할 생성
CREATE ROLE IF NOT EXISTS cortex_user_role;

-- Snowflake가 제공하는 CORTEX_USER 데이터베이스 역할을 부여
GRANT DATABASE ROLE SNOWFLAKE.CORTEX_USER TO ROLE cortex_user_role;

-- 승인된 사용자에게만 역할 부여
GRANT ROLE cortex_user_role TO USER analyst_kim;
```

- 기본적으로 `SNOWFLAKE.CORTEX_USER` 데이터베이스 역할은 `PUBLIC`에 부여되어 있어
  모든 사용자가 LLM 함수를 쓸 수 있는 경우가 많다.
- 통제를 강화하려면 `PUBLIC`에서 이 역할을 **회수(REVOKE)** 한 뒤, 전용 역할을 통해서만 부여한다.

```sql
-- (선택) 전체 개방을 닫고 전용 역할로만 허용
REVOKE DATABASE ROLE SNOWFLAKE.CORTEX_USER FROM ROLE PUBLIC;
```

> 회수는 조직 전체에 영향을 주므로, 사전 공지와 영향도 검토 후 신중히 적용할 것.
> Cortex Analyst / Agent / Intelligence 등은 별도의 접근·객체 권한 체계를 가지므로
> 각 기능 문서의 권한 요구사항을 함께 확인해야 한다.

# SILVER → GOLD dbt 적재 작업 가이드

> 작성일: 2026-07-03 · 대상: GOLD star schema(DIM 15 + FACT 9) 적재 dbt 파이프라인 개발자
> 선행: SILVER 26개(CRM 21 + GA4 5) 구현 완료(`10_dbt_pipeline/models/silver/`)
> 정본: `03_top-down_gold/08_silver의존.md`(lineage), `03_top-down_gold/06_DDL.sql`(GOLD 물리 스키마), `03_top-down_gold/07_메타.md`(제약)
> 방식: **C-2 — dbt + 프로젝트 내장 커스텀 매크로 (외부 패키지 無, trial EAI 불가 유지)**

---

## 0. 이 가이드의 전제 (먼저 읽을 것)

1. **GOLD 구조 소유권 = `06_DDL.sql`**. GOLD 24 테이블은 이미 DDL로 (예정)생성. dbt는 CRM과 동일하게 **`incremental` + `full_refresh:false`로 INSERT/UPSERT만** 수행, 빈 테이블을 `CREATE OR REPLACE`로 덮지 않는다.
2. **통합은 GOLD FACT에서만**. SILVER는 소스 1:1 유지. cross-source(회원↔GA)는 전부 `DIM_MEMBER_IDENTITY` 브리지 경유(`08_silver의존.md §3`).
3. **미수령 원천은 적재 금지**(P9). ERP·대행사(AGENCY/GADS)·어드민(ADMIN) 기반 GOLD는 **컬럼만 존재** → 데이터 넣지 말 것(§4).
4. **대리키(`_SK`)는 ETL이 생성**(SILVER엔 없음). 생성 전략은 §3.

## 1. 적재 순서 (의존 위상)

`06_DDL.sql` 실행규칙 §2 준수. dbt는 `ref()`로 의존을 선언하면 순서를 자동 해석하나, **개념 순서**는 아래와 같다.

```
[1] DIM_DATE                     (팩트 일자 범위로 캘린더 생성, 소스 무관)
[2] DIM_ORG                      (CRM_ORG 계층 전개 + SCD2)
[3] DIM_MEMBER                   (CRM_MEMBER + STATUS_HIST SCD2)
[4] 나머지 DIM                   (CAMPAIGN·SPONSORSHIP·GA_SOURCE·GA_EVENT·DEVICE·
                                  SERVICE·PAYMENT·REASON·EVENT·MEMBER_IDENTITY)
[5] FACT (9)                     (DIM 대리키 조인으로 SK 해소 후 적재)
```

> FACT는 반드시 **DIM 적재 완료 후**. FACT의 `*_SK` 컬럼은 DIM을 조인해 BK→SK로 치환하기 때문.

## 2. 지금 적재 가능 vs 대기 (원천 현황 기준)

`08_silver의존.md §5-C` 커버리지 반영. **SILVER에 실데이터가 있어야 GOLD도 검증 가능**하다는 점 주의.

### 2-A. 즉시 적재·검증 가능 (SILVER 실데이터 존재)
| GOLD | SILVER 소스 | 비고 |
|------|-------------|------|
| `DIM_MEMBER`(부분) | `CRM_MEMBER`(2,000행 실적재) | REGION/AGE_BAND는 개발·증감 테이블 대기(§5-A) |
| `DIM_GA_SOURCE` | `GA4_TRAFFIC_SOURCE`(447) | |
| `DIM_GA_EVENT` | `GA4_EVENT_DIM`(394) | |
| `DIM_DEVICE` | `GA4_DEVICE`(23) | DEVICE_TYPE 파생(예견이슈 E) |
| `DIM_DATE` | (ETL 생성) | GA4_EVENT 일자 범위로 채움 |
| `FACT_GA_BEHAVIOR`(부분) | `GA4_EVENT`(1,930) | IDENTITY_SK는 GA4_IDENTITY 대기 |

### 2-B. SILVER 모델은 있으나 Bronze 미입고 → 대기
`DIM_ORG·DIM_CAMPAIGN·DIM_SPONSORSHIP·DIM_SERVICE·DIM_PAYMENT·DIM_REASON·DIM_EVENT` 및 `FACT_MEMBER_MONTHLY·FACT_MEMBER_EVENT·FACT_TARGET_DEV·FACT_SERVICE_EVENT·FACT_EVENT_PARTICIPATION` — 대응 CRM SILVER 20개가 Bronze 입고 후 실행되면 적재 가능.

### 2-C. 미수령 원천 → 적재 금지 (§4)
| GOLD | 미수령 레이어 |
|------|--------------|
| `DIM_AD_CREATIVE` | 대행사(Agency) |
| `DIM_BUDGET_ITEM` | ERP |
| `FACT_TARGET_BIZ` | ERP |
| `FACT_BUDGET` | ERP(+대행사 편성비) |
| `FACT_AD_PERFORMANCE`(GA_CONV_* 외) | 대행사·GADS |
| `FACT_SERVICE_EVENT.APP_PUSH_*` | 어드민 |
| `FMM.INBOUND_CALL_CNT·TS_CALL_CNT` | 비-CRM 수기 |

> `DIM_MEMBER_IDENTITY`는 `GA4_IDENTITY`(현재 `enabled=false`, Q1 행매칭 실증 대기)에 의존 → **브리지 활성 전까지 FACT_GA_BEHAVIOR.IDENTITY_SK 미해소**.

## 3. 대리키(`_SK`) 생성 전략 — 핵심 결정 필요

SILVER엔 `_SK`가 없다. GOLD DIM PK(`ORG_SK`·`MEMBER_SK` 등 `NUMBER(38,0)`)를 dbt가 만들어야 한다. trial(dbt_utils 불가)에서 **외부 패키지 없이** 아래 중 택1:

| 방식 | 구현 | 장단점 |
|------|------|--------|
| **A. 해시 대리키**(권장) | `HASH(BK컬럼들)` 또는 `SHA1_BINARY`→NUMBER 파생 매크로 | 재현성·병렬안전. `NUMBER(38,0)` 폭 필요. SCD2는 (BK+EFFECTIVE_FROM) 해시 |
| B. Snowflake SEQUENCE | `CREATE SEQUENCE` + `.NEXTVAL` | 순번 깔끔하나 재빌드 시 값 변동, merge와 궁합 주의 |
| C. `ROW_NUMBER()` | 윈도우 순번 | full-refresh 전용, 증분과 충돌 → 비권장 |

**권고: 방식 A(해시)**. 커스텀 매크로로 `dbt_utils.generate_surrogate_key` 대체:
```sql
-- macros/gold_surrogate_key.sql (dbt_utils 대체)
{% macro gold_sk(cols) -%}
    ABS(HASH({{ cols | join(', ') }}))
{%- endmacro %}
```
> SCD2 DIM은 `gold_sk([BK, 'EFFECTIVE_FROM'])`로 버전별 유일 SK 생성. FACT의 `*_SK`는 해당 DIM을 `ref()`로 조인해 BK→SK 치환.

## 4. SCD2 차원 처리 (DIM_ORG · DIM_MEMBER)

SILVER는 **현재상태**만 보유(`CRM_ORG`) 또는 이력 별도(`CRM_MEMBER_STATUS_HIST`). GOLD DIM_ORG/DIM_MEMBER는 SCD2(EFFECTIVE_FROM/TO·IS_CURRENT).

- **DIM_MEMBER**: 상태 이력은 이미 SILVER `CRM_MEMBER_STATUS_HIST`가 SCD2로 산출(EFFECTIVE_FROM/TO·IS_CURRENT) → GOLD는 이를 회원속성과 결합. 나머지 속성(SCD1성)은 `CRM_MEMBER` 현재값.
- **DIM_ORG**: `CRM_ORG`는 현재상태만 → SCD2는 GOLD ETL이 생성. 초기 적재는 IS_CURRENT=TRUE·EFFECTIVE_FROM=최초적재일 단일버전으로 시작, 변경 감지 시 버전 추가.
- **구현 옵션**: dbt `snapshot`(권장 표준) vs `incremental` 수동 SCD2. `snapshot`은 source freshness/전용 스키마 필요 → **GOLD DDL이 구조를 소유하는 본 프로젝트 원칙과 상충** → **`incremental` + merge로 수동 SCD2 권장**(신규/변경행 close-out + insert).

## 5. 제안 폴더 구조 & materialized

```
models/
  silver/               # (기존) CRM 21 + GA4 5
  gold/
    dim/
      DIM_DATE.sql              materialized=incremental (또는 table 1회)
      DIM_ORG.sql               incremental (수동 SCD2)
      DIM_MEMBER.sql            incremental (수동 SCD2)
      DIM_CAMPAIGN.sql          incremental
      DIM_SPONSORSHIP.sql       incremental
      DIM_GA_SOURCE.sql         incremental
      DIM_GA_EVENT.sql          incremental
      DIM_DEVICE.sql            incremental
      DIM_SERVICE.sql           incremental
      DIM_PAYMENT.sql           incremental
      DIM_REASON.sql            incremental
      DIM_EVENT.sql             incremental
      DIM_MEMBER_IDENTITY.sql   incremental  (enabled=false, GA4_IDENTITY 대기)
      # DIM_AD_CREATIVE / DIM_BUDGET_ITEM → 미수령(생성 보류)
    fact/
      FACT_GA_BEHAVIOR.sql      incremental (merge, 일 grain)
      FACT_MEMBER_MONTHLY.sql   incremental (merge, MONTH_KEY grain)
      FACT_MEMBER_EVENT.sql     incremental
      FACT_TARGET_DEV.sql       incremental
      FACT_SERVICE_EVENT.sql    incremental
      FACT_EVENT_PARTICIPATION.sql incremental
      # FACT_TARGET_BIZ / FACT_BUDGET / FACT_AD_PERFORMANCE → 미수령(보류)
```

`dbt_project.yml` 추가:
```yaml
models:
  gn_dw_silver:
    gold:
      +database: GN_DW
      +schema: GOLD
      +materialized: incremental
      +incremental_strategy: merge
      +full_refresh: false          # GOLD DDL 구조 보호(CRM과 동일 원칙)
      +on_schema_change: append_new_columns
```

## 6. 필요한 커스텀 매크로

| 매크로 | 용도 |
|--------|------|
| `gold_sk(cols)` | 해시 대리키 생성(§3) — `dbt_utils.generate_surrogate_key` 대체 |
| `gold_meta(src_system)` | GOLD 공통 감사 4컬럼(DW_SOURCE_SYSTEM·DW_LOAD_TS·DW_UPDATE_TS·DW_BATCH_ID) |
| `month_key(date_col)` | `YYYYMM` 파생(월팩트 conform) — `TO_CHAR(d,'YYYYMM')::NUMBER` |
| `date_sk(date_col)` | `YYYYMMDD`→`NUMBER(8,0)` (DIM_DATE 조인키) |

> 기존 `silver_helpers.sql`의 `clean_str`·`dw_meta`는 SILVER 전용. GOLD는 `gold_meta`(DW_UPDATE_TS 포함) 별도 정의.

## 7. FACT 적재 패턴 (BK→SK 치환)

모든 FACT는 measure를 SILVER에서 가져오고, 차원 외래키는 **DIM을 `ref()` 조인해 SK로 치환**한다.

```sql
-- 예: FACT_GA_BEHAVIOR (일 grain, GA4_EVENT 기반)
{{ config(materialized='incremental', incremental_strategy='merge',
          unique_key=['DATE_SK','IDENTITY_SK','GA_EVENT_SK','GA_SOURCE_SK','DEVICE_SK','PAGE_PATH']) }}

with e as ( select * from {{ ref('GA4_EVENT') }} )
select
    {{ date_sk('e.EVENT_DT') }}                    as DATE_SK,
    d.DEVICE_SK,                                    -- DIM_DEVICE 조인
    s.GA_SOURCE_SK,                                 -- DIM_GA_SOURCE 조인
    ev.GA_EVENT_SK,                                 -- DIM_GA_EVENT 조인
    -- id.IDENTITY_SK,   ⚠️ GA4_IDENTITY 활성 후
    e.PAGE_LOCATION                                 as PAGE_PATH,
    COUNT(DISTINCT e.GA_SESSION_ID)                 as SESSION_CNT,
    COUNT(*)                                        as EVENT_CNT,
    {{ gold_meta('GA4') }}
from e
left join {{ ref('DIM_DEVICE') }}     d  on d.DEVICE_TYPE = e.DEVICE_TYPE
left join {{ ref('DIM_GA_SOURCE') }}  s  on s.UTM_SOURCE = e.UTM_SOURCE and s.UTM_MEDIUM = e.UTM_MEDIUM
left join {{ ref('DIM_GA_EVENT') }}   ev on ev.EVENT_NAME = e.EVENT_NAME
group by 1,2,3,4,5
```
> **비/준가산 지표 주의**(06_DDL §6): `BOUNCE_RATE·AVG_SESSION_DURATION` 등은 grain 값 — 상위 재합산 금지.

## 8. 미해소 매핑 (적재 전 확인 — 추정 금지)

`08_silver의존.md §5-A` 잔여 4건. GOLD 적재 시 그대로 승계:

| GOLD | 미해소 | 조치 |
|------|--------|------|
| `DIM_CAMPAIGN.ORG_SK` | 캠페인↔조직 연결원천 미확정 | 개발테이블 `CMPGN_CD`+`ACT_DEPT_CD` 병존으로 파생 가능하나 '주관조직' 정의(O10) 대기 |
| `DIM_SERVICE.SEND_TYPE_L/M/S·SUBTYPE` | `SND_REQ_MST` 대/중/소 컬럼 의미 검수 대기 | 코드체계 확인 후 |
| `DIM_EVENT.EVENT_CATEGORY·APPLY_CHANNEL` | 원천/파생 규칙 미정 | ADMIN(A-10) 대기 |
| `DIM_PAYMENT.FEE_TYPE` | 회비유형 정기/일시 이중표현 | 설계결정 대기 |
| `DIM_MEMBER.REGION·AGE_BAND` | 소스 확정(개발·증감 AREA_CD CM018·AGE) | SILVER `CRM_MEMBER_DEV`/`AMT_CHANGE` projection 편입만 잔여 |

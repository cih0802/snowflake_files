# 변환 로직 성능 타당성 검토서 (대용량 처리 관점)

> **목적**: 복잡한 조인(Join)·윈도우 함수 연산이 대용량 데이터에서 **성능 저하·과도 비용·동시성 충돌**을 유발하지 않는 스키마/변환 구조인지 검증한다.

## 1. 변환 로직 유형별 성능 리스크

### 1-1. 🔴 스캐폴드 팩트 — 매 run 전량 재적재 (D1 FROZEN)
- **현 구조**: 팩트 6종(FGA·FSE·FMM·FME·FEP·FTD)이 `incremental + append + pre-hook TRUNCATE + full_refresh:false`. 매 실행마다 TRUNCATE 후 **전량 재적재**(멱등·구조보존).
- **리스크**: 3,800만 행 팩트를 매 run 전량 rewrite → **웨어하우스 비용·실행시간 증가**. 데이터 성장 시 선형 악화.
- **판정**: 🔴 **마감 전 재평가 필수(D1 FROZEN)**. 볼륨·비용 측정 후 진성 `incremental merge`(unique_key=활성 SK, 증분만 upsert)로 전환 여부 결정.
- **왜 지금 append+TRUNCATE인가**: `table`(CTAS)은 06_DDL의 FK/타입/COMMENT를 드롭하여 폐기됨. append+TRUNCATE가 구조보존+멱등의 절충. (진단: 이슈 10 §3, 30 §2)

### 1-2. 🟡 SCD2 시점조인 + WIDE fan-out 방지 윈도우 함수
- **SCD2 재구성**: `DIM_MEMBER`는 `TH_MM_FDRM_MBER_STNG_DTLS`(750만) 상태이력을 시점조인해 792만 버전행 생성. 시점조인(`EFFECTIVE_FROM/TO BETWEEN`)은 **범위조인(range join)** 으로 대용량에서 비싼 연산 후보.
- **WIDE fan-out 방지**: WIDE 뷰가 SCD2 회원차원을 조인할 때 `WHERE IS_CURRENT=TRUE` + `qualify ROW_NUMBER() OVER (PARTITION BY MEMBER_DK ...)` → 회원당 현재 1행. **792만 파티션 윈도우 연산**이 WIDE 조회마다 발생(뷰=물리 0).
- **리스크**: WIDE_MEMBER_MONTHLY 등은 3,779만 팩트 × SCD2 window를 **런타임에** 계산 → BI 반복 조회 시 반복 비용.
- **판정**: 🟡 정합성은 올바름(fan-out 차단). 대용량 반복조회 시 **WIDE 일부를 물리화(materialized table/dynamic table) 검토** 또는 회원 현재버전 서브셋을 사전 물리화.

### 1-3. 🟡 merge 차원 stale (P7)
- **현상**: GOLD 차원은 `incremental merge`(pre-hook TRUNCATE 없음) → 상류 grain/로직 변경 시 **옛 행 잔존**(DIM_GA_EVENT 2,842→2,846 사고, 2026-07-16 복원).
- **판정**: 🟡 완전 재산출 차원은 **append+pre-hook TRUNCATE(fact 패턴)** 전환 시 잔존행 원천 차단(`DIM_MEMBER` SCD2가 이 패턴 채택). grain 변경 이력 있는 merge 차원 동일 검토 권장.

### 1-4. 🔴 identity pseudo-grain 성장 (G-5 트리거)
- **현상**: `GA4_IDENTITY`·`IDENTITY_MEMBER_XREF`는 grain=`user_pseudo_id`. ITP 7일 만료·기기이탈로 pseudo 지속 증가 → 두 모델 계속 성장. 현재 append+TRUNCATE+윈도우라 **매 실행 전기간 재스캔**.
- **리스크**: G-5(전기간 입고)·상시운영 전환 시 **매 실행 전기간 재스캔 = 비용 급증**.
- **판정**: 🔴 **G-5 시점 조치**: `incremental merge`(unique_key=`USER_PSEUDO_ID`) + TRUNCATE 제거 → 신규 pseudo만 upsert. 성장 시 `CLUSTER BY`(member/event_date)로 조인 프루닝. 모델별 config 오버라이드로 격리.
- **주의**: GOLD `DIM_MEMBER_IDENTITY`·FACT(`IDENTITY_SK`)는 **회원 grain이라 bounded**(안전). 조치 대상은 SILVER pseudo-grain 2모델뿐.

### 1-5. 🟡 Unknown 센티넬(SK=0) 집계 스큐
- **현상**: 미매칭 차원키를 `COALESCE(...,0)`로 Unknown(0) 라우팅. 결손 규모가 큰 경우(예: 이슈 E 참여 23%·GA identity 미매칭 대부분) **0 버킷에 대량 집중**.
- **리스크**: `GROUP BY <dim>_SK` 집계 시 0 버킷 데이터 스큐 → 파티션 불균형·집계 편중.
- **판정**: 🟡 정합성 안전(조인 유실 방지). 분석 시 Unknown 버킷 분리 인지 필요. 원천 입고로 결손 해소 시 자연 완화.

### 1-6. 🟡 조인 fan-out (회원 다중후원 O8)
- **현상**: `FSE`·`FEP`의 캠페인·후원사업 귀속을 **직접 FK**로 잠정확정(O8). 회원이 다중 후원사업 보유 시 조인 fan-out 우려.
- **판정**: 🟡 **회원 다중후원 카디널리티 검증**이 게이트. P3 진단패턴(중복키=0 & 조인후≈base → N:1 LEFT JOIN 스파인 보존) 적용 필요.

## 2. 교착(Deadlock)·동시성 관점 (Snowflake 특성)

- **전통 RDBMS deadlock 없음**: Snowflake는 MVCC(스냅샷 격리)·마이크로파티션 불변 구조 → 행 잠금 기반 교착은 발생하지 않음. dbt run은 순차/DAG 실행이라 상호 교착 위험 낮음.
- **실제 리스크 = 쓰기 충돌·비용**:
  - append+TRUNCATE 팩트는 매 run **전량 파티션 rewrite** → 동시 실행 시 같은 테이블 write 충돌 가능(운영은 단일 오케스트레이션으로 회피).
  - merge 차원은 동일 키 동시 갱신 시 write conflict(dbt 순차라 무해).
- **판정**: 🟢 현 단일 오케스트레이션(dbt DAG) 하에서 교착·동시성 이슈 없음. 병렬 스케줄 도입 시 팩트 rewrite 겹침만 회피하면 됨.

## 4. 종합 판정

- **현재 규모(팩트 최대 3,800만)에서는 build green·실행 가능** — 구조적 성능 결함 없음.
- **마감/상시운영/전기간(G-5) 전환 시 재평가 필수 항목 2건**: ① 스캐폴드 팩트 전량 재적재(D1) · ② identity pseudo-grain 전기간 재스캔(G-5).
- **교착은 Snowflake 특성상 비해당**. 동시성은 단일 오케스트레이션으로 관리됨.
- **정합성 관점 안전장치는 이미 적용됨**: SCD2 fan-out 차단(IS_CURRENT+qualify)·Unknown 센티넬(SK=0)·N:1 LEFT JOIN 스파인 보존.

## 1. 대용량 조인(Join) 타당성

### 1-1. WIDE 뷰 = 비물질화(view) 팩트×차원 LEFT JOIN
- 모든 `WIDE_*`는 `materialized='view'` → **조회 시점마다** 팩트(수천만 행)에 DIM을 LEFT JOIN.
- 대부분 DIM은 소형(수만 행 이하) → Snowflake가 **broadcast join**으로 처리, 비용 낮음.
- **주의 조인**: `WIDE_MEMBER_MONTHLY`(37.79M) ⨯ `DIM_MEMBER`(7.93M) on `MEMBER_DK`.
  - DIM_MEMBER는 broadcast 하기엔 큰 편(7.93M) → **hash join** 유력. 조인키 `MEMBER_DK`(VARCHAR(10)) 해시 분산 양호.
  - 위험: 팩트·DIM 동시 대형 → 셔플·**메모리 spill** 가능(웨어하우스 과소 시).
- **판정**: 구조상 치명적 문제 없음. 단 WIDE_MEMBER_MONTHLY는 spill 모니터링 대상(§7).

### 1-2. SK=0 Unknown 라우팅
- 팩트 FK는 `COALESCE(...,0)`로 미매칭을 0에 몰아넣음 → **0 값 skew**(핫 파티션) 가능.
- 소형 DIM 조인이라 실질 영향 작으나, 0 비중 높은 팩트(예: 광고성과 캠페인 미매핑)는 skew 관측 권장.

## 2. 윈도우 함수 연산 타당성

### 2-1. 빌드 시(build-time) — `DIM_MEMBER` SCD2
```
qualify row_number() over (partition by MBER_NO, EFFECTIVE_FROM::DATE order by SER_NO desc) = 1   -- 동일시점 축약
lead(EFF_FROM) over (partition by MBER_NO order by EFF_FROM)                                       -- SCD2 구간 재계산
```
- 입력 STATUS_HIST 7.5M을 `MBER_NO`로 파티션 → **파티션당 행수 작음**(회원당 평균 4.5·최대 218버전). 파티션 내 정렬 비용 낮음.
- 1회성 빌드(월배치)라 상시 부하 아님. **판정: 타당**. 대형 정렬이지만 파티션 분할로 spill 위험 제한적.

### 2-2. 조회 시(query-time) — WIDE 뷰 내부 윈도우 ⚠️ 핵심 주의
```
-- WIDE_MEMBER_MONTHLY 내부
left join ( select ... from DIM_MEMBER where IS_CURRENT=TRUE
            qualify ROW_NUMBER() OVER (PARTITION BY MEMBER_DK ORDER BY EFFECTIVE_FROM DESC ...) = 1 ) m
```
- 이 윈도우는 **뷰가 조회될 때마다** DIM_MEMBER 전체(현재행)에 대해 실행됨(비물질화).
- 현재행 중복 제거 목적의 방어 로직(SCD2 fan-out 방지)이나, **매 쿼리 반복 비용**.
- **위험**: 대시보드 동시 조회가 많으면 반복 윈도우+대형 조인이 웨어하우스 부하로 누적.
- **권고**: (a) BI 동시성/지연이 문제되면 `WIDE_MEMBER_MONTHLY`를 물질화(table/dynamic table)하거나, (b) DIM_MEMBER에 현재행 전용 뷰(`DIM_MEMBER_CURRENT`)를 별도 물질화해 뷰 내부 윈도우 제거.

## 3. 교착(Deadlock)·동시성 타당성

- **Snowflake는 MVCC** — 읽기는 락을 잡지 않음(스냅샷 격리). 전통적 read-write deadlock 없음.
- **DML 동시성**: 같은 테이블 동시 `MERGE`/`UPDATE`/`DELETE`는 **파티션 수준 락**으로 직렬화(교착이 아니라 대기·재시도). 
  - DIM(merge, unique_key)과 FACT(append+pre-hook TRUNCATE)는 **dbt 단일 run 내 모델별 순차 실행** → 자기 테이블 동시 DML 없음. **교착 위험 없음**.
- **주의**: FACT 패턴이 `pre-hook: TRUNCATE + append`이므로, **TRUNCATE와 조회가 겹치면** 조회는 직전 커밋 스냅샷을 보되(정합), 적재 중 부분노출은 없음(트랜잭션). 단 **적재 창(window) 중 WIDE 조회는 빈/구 데이터**가 될 수 있어 배치 시간대 공지 권장.
- **판정**: 교착 위험 구조적으로 낮음. 운영은 "적재 창 ↔ BI 조회" 시간 분리 권고.

## 4. 마이크로파티션·프루닝·클러스터링

- 명시적 `cluster_by` 미설정. FACT는 append 순서(대체로 시간순) → `MONTH_KEY`/`DATE_SK` 필터 시 **자연 프루닝** 기대.
- 위험: 재적재(TRUNCATE+append)가 소스 순서를 바꾸면 클러스터링 깊이 악화 가능.
- **권고**: 대표 조회 필터(`MONTH_KEY`, `DATE_SK`)에 대한 **클러스터링 깊이 모니터링**(`SYSTEM$CLUSTERING_INFORMATION`). 악화 시 대형 팩트에 한해 `cluster_by=(MONTH_KEY)` 검토(비용 대비 효과 확인 후).

## 5. Spill(디스크 유출) 위험

- 유발원: §1-1 대형 hash join, §2-2 query-time 윈도우 정렬, 대형 GROUP BY.
- **권고**: BI용 웨어하우스에서 `BYTES_SPILLED_TO_LOCAL/REMOTE_STORAGE` 모니터링. remote spill 관측 시 웨어하우스 1단계 상향 또는 WIDE 물질화(§2-2).

## 6. 종합 판정

| 영역 | 판정 | 핵심 조치 |
|---|---|---|
| 대용량 조인 | 🟢 구조 타당 | WIDE_MEMBER_MONTHLY spill 모니터링 |
| 윈도우(빌드) | 🟢 타당 | — |
| 윈도우(조회) | 🟡 주의 | WIDE 물질화 또는 DIM_MEMBER_CURRENT 분리 검토 |
| 교착/동시성 | 🟢 위험 낮음 | 적재 창↔조회 시간 분리 |
| 프루닝/클러스터링 | 🟡 관측 필요 | 클러스터링 깊이 모니터링 |
| Spill | 🟡 관측 필요 | spill 지표 모니터링·웨어하우스 사이징 |

## A. GA4_EVENT 증분 최초적재 함정 🔴

**현상**: `SILVER_DDL_20260702.sql`이 `GA4_EVENT` 빈 테이블을 **선생성**해 둔 상태에서 일반 `dbt run`을 하면, dbt `is_incremental()`이 **첫 실행부터 true**가 된다(대상 relation이 이미 존재하므로). 그러면 증분 분기(`D-3 ~ D-1`)를 타는데, 보유 shard `EVENTS_20260501`(과거일)은 이 창에 안 들어와 **0행 적재**된다.

**근거**: dbt `is_incremental()` = (모델 incremental) AND (대상 테이블 존재) AND (NOT --full-refresh). DDL 선생성이 2번 조건을 첫 run부터 충족.

**해결**: `dbt_project.yml`에 GA4만 `+full_refresh: true` 부여 + **최초/백필은 반드시**
```
dbt run --select silver.ga4 --full-refresh
```
→ `else` 분기(전체 shard UNION)로 과거분 포함 적재. 실측 결과 GA4_EVENT 1,930행 정상.

**잔여 리스크**: 신규 환경 배포·재해복구 시 이 규칙을 모르면 **조용히 0행**. → `MIGRATION_GUIDE.md` 및 배포 런북에 "GA4 최초적재 = --full-refresh" 고정 기재 필요.

## B. GA4 차원 `table` 재생성 → DDL 구조 소유권 이전 🟡

**현상**: `GA4_TRAFFIC_SOURCE`·`GA4_EVENT_DIM`·`GA4_DEVICE`는 `materialized='table'` → 매 run `CREATE OR REPLACE` → DDL이 정의한 **PK·타입·길이를 dbt SELECT 결과가 덮어쓴다.**

**CRM과의 비대칭**: CRM 21개는 `incremental` + `full_refresh:false`로 "**DDL이 구조 소유**" 원칙(dbt_project.yml §materialization 방침)을 지켰다. GA4 차원은 설계결정서가 명시 허용("Silver 26 DDL … 또는 dbt materialized로 대체 생성")하나, 결과적으로 **구조 소유권이 dbt SELECT로 이전**된다.

**리스크**: SELECT의 캐스팅이 DDL과 미세하게 어긋나면(예: DDL은 `VARCHAR` 무길이, SELECT는 파생 표현식 길이) GOLD 계약과 대조 시 불일치 가능. → 차원 3개는 배포 후 `DESCRIBE TABLE`로 DDL 대조 1회 권장.

## C. GA4 모델이 dbt `source()` 우회 → lineage 갭 🟡

**현상**: `ga4_union_shards` 매크로가 `run_query`로 shard 목록을 뽑아 **raw FQN**(`GN_DW.BRONZE_GA4.EVENTS_...`)을 직접 참조한다. dbt `source()`를 쓰지 않으므로 **manifest/`dbt docs`에 BRONZE_GA4 → GA4 엣지가 안 잡힌다.**

**영향**: `dbterd` ERD에서 GA4 상류 리니지 누락. `08_silver의존.md`의 수기 lineage로 보완 중이나, 자동 문서화 범위 밖.

**불가피성**: date-shard가 동적이라 정적 `source()` 선언 불가(설계상 트레이드오프). → GA4 상류 리니지는 **수기 문서가 정본**임을 명시.

## D. GA4_EVENT 복합 PK 유일성, 단일 shard만 검증 🔴

**현상**: merge `unique_key = [USER_PSEUDO_ID, EVENT_TIMESTAMP, EVENT_NAME, BATCH_ORDERING_ID]`. 현재 `EVENTS_20260501` 1개 shard에서만 1,930 = 1,930 유일 확인(`GA4_EVENT` DDL 주석의 GA-1과 동일 미검증 상태).

**리스크**: 다중 shard/전량에서 이 4키가 유일하지 않으면, merge가 충돌행 중 **임의 1행만 남기고 조용히 버린다**(silent loss). GA4 export 특성상 동일 pseudo_id가 같은 microsec에 동일 event를 여러 번 남기는 경우가 있음.

**권고**: 다중 shard 적재 직후 아래로 유일성 검증 → 실패 시 키에 `event_bundle_sequence_id` 또는 `batch_event_index` 추가 검토.
```sql
SELECT USER_PSEUDO_ID,EVENT_TIMESTAMP,EVENT_NAME,BATCH_ORDERING_ID, COUNT(*)
FROM GN_DW.SILVER.GA4_EVENT GROUP BY 1,2,3,4 HAVING COUNT(*)>1;
```

## E. DEVICE_TYPE(PC/M/APP) 파생 휴리스틱 가정 🟡

**현상**: `CASE WHEN platform IN ('ANDROID','IOS') THEN 'APP' WHEN device:category IN ('mobile','tablet') THEN 'M' ELSE 'PC' END`. `device.category`의 실제 도메인 값을 전량 확인하지 않음.

**리스크**: category에 `desktop`/`smart tv`/NULL 외 예상 못한 값이 오면 **PC로 오분류**. GOLD `DIM_DEVICE.DEVICE_TYPE`가 이 파생에 직결 → 디바이스별 지표 왜곡.

**권고**: 다중 shard 적재 후 `SELECT DISTINCT device:category, platform`로 도메인 확정 → CASE 보정.

**권고**: Bronze 스키마별 입고 시 해당 모델 `dbt run` + PK/NULL 검증을 CRM_MEMBER와 동일 절차로.

## G. CRM_CODE 라벨 백필 교차의존 🟡

**현상**: 아래 `_NM`(라벨) 컬럼들이 현재 **NULL 보존** 상태 — `CRM_CODE` 적재 후 채워야 함.
- `CRM_MEMBER.MBER_DIV_NM`·`MBER_STAT_NM`
- `CRM_MEMBER_STATUS_HIST.BF_STAT_NM`·`CHN_STAT_NM`
- `CRM_MEMBER_DISCONTINUE.DSCNTC_RSN_NM`
- `CRM_PAYMENT_METHOD.SETLE_NM`

**미설계 지점**: 라벨 채우기를 (1) 각 모델에서 `CRM_CODE`를 `ref()` 조인으로 인라인할지, (2) 별도 post-hook/후속 모델로 UPDATE할지 **결정 안 됨**. 현재는 코드값만 정제하고 라벨은 GOLD 조인으로 미룬 상태.

**권고**: GOLD에서 `CRM_CODE` 조인으로 라벨 해소하는 편이 SILVER 재빌드 의존을 줄임(설계결정 필요). 결정 전까지 `_NM` = NULL 유지가 안전.

## H. `full_refresh` 이원화 운영 혼선 🟢

**현상**: `silver`(CRM) = `full_refresh:false`, `silver.ga4` = `full_refresh:true`. 팀원이 전체 `dbt run --full-refresh` 실행 시 **CRM은 무시(DDL 보호)·GA4는 CREATE OR REPLACE**로 동작이 갈린다.

**권고**: 이 비대칭이 **의도된 설계**임을 런북에 1줄 명기(CRM=DDL 소유/GA4=dbt 소유). 오해로 인한 "왜 CRM은 재생성 안 되나" 문의 예방.

## I. `clean_str`의 `'-'`·`'NULL'` 치환 부작용 🟢

**현상**: `clean_str` = `NULLIF(NULLIF(NULLIF(TRIM(x),''),'NULL'),'-')`. 값이 정당하게 `'-'` 또는 문자열 `'NULL'`인 컬럼에 적용되면 **유효값이 NULL로 손실**.

**리스크**: CRM 키/코드 컬럼 중 `'-'`가 유효 구분자거나 실제 값인 경우. 현재 회원 2개 테이블에선 무해 확인.

**권고**: Bronze 입고 시 `clean_str` 적용 컬럼별로 `'-'`/`'NULL'` 원본 빈도 점검 후, 키 컬럼은 `TRIM`만 적용하는 변형 매크로 검토.

### K. 팩트 SK 센티넬 0 → 차원 unknown 멤버 필수 🔴
`FACT_GA_BEHAVIOR` 전 1,243행에서 `IDENTITY_SK=0`·`CAMPAIGN_SK=0`(센티넬). GA 차원(SOURCE/EVENT/DEVICE)은 실측상 SK=0 고아 **0건**(모든 NULL 조합까지 차원이 커버). 그러나 `DIM_MEMBER_IDENTITY`·`DIM_CAMPAIGN` 등 **센티넬 대상 차원에 SK=0 행이 없으면 BI inner join 시 팩트 행 전량 유실**. → **8개 scaffold 차원(ORG·CAMPAIGN·SPONSORSHIP·SERVICE·PAYMENT·REASON·EVENT·MEMBER_IDENTITY)에 `(미매핑)` unknown(SK=0) 멤버 UNION 적용**. 후속: 신규 차원 추가 시 동일 패턴 필수.

### L. 로직 변경 시 TRUNCATE 필수 🔴
GOLD/SILVER 모두 `full_refresh:false`(DDL 구조 보호)라 `--full-refresh`가 무시됨. **모델 SQL 로직(SK 산식·grain·조인)을 바꾸면 기존 행이 merge로 안 지워지고 누적**된다. 실측: DIM_GA_SOURCE grain 변경 후 FACT_GA_BEHAVIOR 12,202→13,445 누적 → `TRUNCATE TABLE` 후 재적재로 1,243 정상화. **운영 규칙: 로직 변경 = 대상 테이블 TRUNCATE 후 `dbt run`**. (초기 적재/컬럼추가는 merge로 안전, 로직변경만 해당)

### M. DIM_GA_SOURCE grain 축소 🟡
`06_DDL.sql`의 DIM_GA_SOURCE 의도 grain=source/medium/campaign/content/term. 그러나 `GA4_EVENT`(팩트 소스)가 content/term 컬럼을 보유하지 않아 조인 불가 → 구현 grain=(source, medium)로 축소(content/term/source_medium은 MAX 대표값 보조). 미조정 시 팩트 팬아웃(실측 6x) 발생. GOLD 계약 문서와 대조 필요. 해소하려면 GA4_EVENT(silver)에 content/term 투영 추가 → SILVER DDL 동반 수정.

### N. `dbt run` 무선택 = 전체 실패 🟡
CRM 20개 silver + GOLD scaffold 13개가 미입고 Bronze를 참조 → 무선택 `dbt run`은 이들에서 실패. **안전 실행 커맨드**:
- 실데이터: `dbt run --select CRM_MEMBER` / `dbt run --select silver.ga4 --full-refresh` / `dbt run --select tag:gold_ready`
- 입고 후: `dbt run --select silver.crm` → `dbt run --select tag:gold_pending`

### O. GOLD 스캐폴드 measure placeholder 🟡
scaffold FACT(MEMBER_MONTHLY·MEMBER_EVENT·SERVICE_EVENT·EVENT_PARTICIPATION)의 활동/증감/누적/밴드/성과 지표는 대부분 0/NULL 골격. 키·구조·차원조인만 정합. 회원상태·개발·증감 통합 산식은 Bronze 입고 후 구현. FACT_TARGET_DEV만 실질 매핑 완비.

### P. 팩트-회원 자연키 조인(SCD2 팬아웃) 🟡
GOLD 팩트는 MEMBER를 `MEMBER_SK`가 아닌 `MEMBER_DK`(자연키)로 조인(06_DDL 설계). `DIM_MEMBER`가 SCD2로 다버전이 되면 `MEMBER_DK`가 중복 → 팩트 조인 시 팬아웃. **BI/조인은 `DIM_MEMBER.IS_CURRENT=TRUE` 필터 필수**(또는 이벤트일자 range 조인).

### Q. DIM_DATE 스파인 공백 가드 🟢
`DIM_DATE`는 `GA4_EVENT`의 MIN/MAX 일자로 스파인 생성. GA4_EVENT가 비면 bounds NULL → DATE_SK NULL(PK NOT NULL 위반)로 실패. 현재 데이터 존재해 무해. 후속: `where d0 is not null` 가드 또는 고정 캘린더 범위 권장.

### R. unknown 멤버 적용 범위 🟢
GA 실적재 차원(SOURCE/EVENT/DEVICE)은 고아 0이라 unknown 멤버 미적용. Kimball 일관성(모든 차원에 unknown) 표준화 시 GA 차원에도 추가하고 gold_ready 재적재(TRUNCATE 후) 검토.

### S. GA4_IDENTITY 행매칭 0건 — 샘플 분리(Q1 실증 결과) 🔴
**2026-07-03 실측**: GA `USER_ID`(distinct 578) ↔ CRM `MEMBER_DK` **직접 매칭 0건**. 포맷은 호환(양측 FDRM 7자리 zero-padded, ONCE `S`+8자리)이고 매핑 로직도 정확하나 **값이 전혀 겹치지 않음**.

| 그룹 | 범위 | 건수 |
|------|------|------|
| GA_FDRM | 0025473 ~ 1985128 (산재) | 983 |
| CRM_FDRM | 0905623 ~ 0906638 (연속 샘플) | 1,000 |
| GA_ONCE | S00189796 ~ S00299464 | 16 |
| CRM_ONCE | S00000001 ~ S00265910 | 1,000 |

**원인**: CRM(1,000행)·GA(2,000행)가 **독립 추출된 샘플**이라 키 교집합이 없음(로직/포맷 문제 아님).

**결론**: 현재 샘플로 GA4_IDENTITY 브리지 **검증·적재 불가**. 활성화 시 `FACT_GA_BEHAVIOR.IDENTITY_SK` 전건 unknown(0) → 가치 없음. **`GA4_IDENTITY`·`DIM_MEMBER_IDENTITY` 비활성 유지가 올바름.**

**해소 조건(택1)**:
1. CRM·GA를 **동일 모집단에서 정합 추출**한 데이터로 재적재 후 매칭률 재측정(권장).
2. 전량(비샘플) 적재 시 자연 매칭 확인.
3. GA-only 신원도 `DIM_MEMBER_IDENTITY`에 포함하는 설계로 변경(CRM 미존재 GA 회원도 SK 부여) → FACT가 GA 신원에 조인 가능. 단 GOLD DDL/의미 재검토 필요.

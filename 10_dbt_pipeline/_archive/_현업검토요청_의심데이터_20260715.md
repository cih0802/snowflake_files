# [GN_DW 현업 검토 요청 — 의심 데이터값 · 미충전 컬럼]

## 목적 및 배경
순서 9~9-B 세션에서 GOLD 팩트/차원을 빌드·검증하며, **파이프라인이 자동으로 "무효/쓰레기"로 판단해 별도 처리(0 라우팅·경고 강등·제외)한 값**들이 있습니다.
이 판단은 **형식(범위·자릿수·타입) 기준의 보수적 추정**이며, 실제로는 **현업 관점에서 유효한 값일 수 있습니다**(예: 특수 인코딩, 과거/미래 데이터, 아직 미입고된 원천).

⚠️ **요청**: 아래 각 항목이 (A) 실제 무효/오류인지, (B) 유효한데 파이프라인이 잘못 배제한 것인지 판정해 주세요. (B)로 확인되면 해당 처리 로직을 되돌리거나 규칙을 수정하겠습니다.

- **아무 것도 삭제하지 않았습니다.** 모든 원본 행은 SILVER/GOLD 에 보존돼 있으며, 처리는 "키를 Unknown(0)으로 라우팅" 또는 "테스트를 경고로 강등"에 한정됩니다(복구 가능).
- 판정 후 "판정" 열에 `유효` / `무효` / `조치필요` 를 기입해 회신 부탁드립니다.

| 구분 | 내용 |
|---|---|

| SILVER | `CRM_PAYMENT_BILLING.MBRFEE_MT` (회비월) |
| GOLD | `FACT_MEMBER_MONTHLY.MONTH_KEY` |
| 의심값 예시 | 최소 `20251` (5자리), 최대 `210103` (2101년 03월), 그 외 월 부분이 13~99인 값 |
| 건수 | 범위밖(199101 미만) 2,040행 + 범위초과(203512 초과) 3행, 이 중 월>12 인 값 1,990행 (총 distinct 약 2,043) |
| 현재 처리 | `MBRFEE_MT` 가 유효 YYYYMM(199101~203512, 월 01~12)이 **아니면 무시** → 납입월(`PAY_DE`)로 대체 → 그것도 없으면 **MONTH_KEY=0 (Unknown월)**. 실측: 최종적으로 0월로 떨어진 행은 53건, 나머지는 납입월로 복구됨 |

**현업 확인 질문**
1. `20251`, `210103` 같은 값은 회비월을 뜻하는 특수 표기(예: 자릿수 누락, 다른 코드체계)인가요, 아니면 입력 오류인가요?
2. 2101년 등 미래 월은 선납/예약 회비로 유효한가요?
3. 무효라면 "납입월로 대체"가 맞는 처리인지, 아니면 별도 규칙(예: 특정 월로 귀속)이 필요한지?

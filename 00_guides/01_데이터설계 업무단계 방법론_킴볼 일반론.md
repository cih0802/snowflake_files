# 데이터 설계 업무 단계 (차원 모델링 기준)

> **문서 읽기 순서: 1단계 → 2단계 → 3단계**
> - **1단계 (현재 문서)**: 킴볼 차원 모델링 방법론 — 설계 원칙·4단계 프로세스·WBS·용어. 벤더 중립 이론.
> - **2단계**: [`모던 엔터프라이즈 데이터 아키텍처 설계 가이드.md`](모던%20엔터프라이즈%20데이터%20아키텍처%20설계%20가이드.md) — 이 방법론을 클라우드 DWH / Modern Data Stack에 적용한 일반 설계 원칙(벤더 중립).
> - **3단계**: [`굿네이버스_snowflake 기반 엔터프라이즈 데이터 아키텍처 설계 가이드.md`](굿네이버스_snowflake%20기반%20엔터프라이즈%20데이터%20아키텍처%20설계%20가이드.md) — 2단계 원칙의 Snowflake(GN_DW) 구현 표준.

차원 모델링(Dimensional Modeling)은 랄프 킴볼(Ralph Kimball)의 방법론에 따라 명확한 4단계 프로세스로 설계됩니다. 핵심은 **비즈니스 요구사항을 분석 가능한 데이터 구조(스타 스키마)로 변환**하는 것입니다. 아래에 설계 4단계, 프로젝트 수행 계획, 실무 주의점을 정리합니다.

> 용어 주의: 이 문서는 **"팩트 테이블 유형"**(트랜잭션/스냅샷 등 구조)과 **"측정값 가산성"**(가산/준가산/비가산)을 구분해서 사용합니다. 둘은 서로 다른 개념입니다(§3-2 참고).

---

## 1. 차원 모델링 설계 4단계

킴볼의 4단계는 아래 **순서대로** 진행하는 것이 원칙입니다. 특히 2단계(그레인) 확정 전에 3·4단계를 확정하면 데이터 레벨 불일치로 모델을 재설계해야 할 수 있습니다.

### 1단계: 비즈니스 프로세스 선택 (Select the Business Process)
- 분석 대상이 되는 **업무 활동(이벤트)**을 결정합니다.
- 조직 부서가 아니라 "측정 가능한 업무 사건"을 고릅니다.
- 예: 상품 판매, 웹사이트 클릭 로그, 자재 입고 등

### 2단계: 그레인 정의 (Declare the Grain) 🌟
- 팩트 테이블의 **한 행(Row)이 정확히 무엇을 의미하는지** 결정합니다.
- 가능한 한 **가장 상세한 원자적 수준(atomic grain)**으로 정의해야, 향후 다양한 집계·드릴다운 요구에 대응할 수 있습니다. (필요 시 원자 팩트 위에 집계 팩트를 별도로 구축)
- 예: "고객이 영수증 하나당 구매한 개별 상품 품목(Line Item) 단위"

### 3단계: 차원 식별 (Identify the Dimensions)
- 그레인이 결정되면, 그 행을 둘러싼 **맥락(Context)**을 정의합니다.
- 주로 언제(When)·어디서(Where)·누가(Who)·무엇을(What)·왜(Why)·어떻게(How)에 해당합니다.
- 모든 차원은 **선언한 그레인과 정합**해야 합니다.
- 예: 일자, 매장, 고객, 상품 차원 등

### 4단계: 팩트(측정값) 식별 (Identify the Facts)
- 비즈니스 프로세스에서 발생하는 **계량적 수치(measure)**를 결정합니다. (수량, 금액 등)
- 2단계에서 정의한 그레인과 **완벽히 일치하는 수준**의 값이어야 합니다.
- 비율·평균·누계 등 **파생 지표는 원칙적으로 원자 값(분자·분모)으로 저장**하고, 계산은 조회 시점(뷰/시맨틱 레이어)에서 수행하는 것을 권장합니다.
- 예: 판매 수량, 매출 금액, 할인 금액 등

## 2. 프로젝트 수행 계획 (WBS 요약)

아래 단계 구성은 표준 흐름이며, **기간(주차)은 프로젝트 규모·데이터 복잡도·원천 상태에 따라 달라지는 예시**입니다.

### Phase 1: 요구사항 정의 및 비즈니스 이해
- **주요 활동**: 현업 인터뷰, 기존 보고서·지표 분석, 핵심 KPI 도출
- **산출물**: 비즈니스 요구사항 정의서, KPI/지표 목록

### Phase 2: 차원 모델링 설계 (고수준)
- **주요 활동**: 4단계 프로세스 적용, 고수준 데이터 모델링, 공유 차원(Conformed Dimension) 식별
- **산출물**:
  - 킴볼 버스 매트릭스 (행=비즈니스 프로세스, 열=공유 차원의 관계 매트릭스)
  - 논리 차원 모델 설계도 (스타 스키마)

### Phase 3: 상세 설계 및 스키마 확정
- **주요 활동**: 차원 속성 상세화, **팩트 테이블 유형 결정**(트랜잭션/주기적 스냅샷/누적 스냅샷), **측정값 가산성 분류**(가산/준가산/비가산), SCD 유형 지정, 데이터 타입 확정, 대리키(Surrogate Key) 설계
- **산출물**: 물리 데이터 모델 설계도(ERD), 데이터 사전, 테이블 정의서

### Phase 4: 검증 및 매핑 계획
- **주요 활동**: 원천(OLTP) → 데이터 웨어하우스 적재 규칙 검토, 그레인·키 무결성 검증, 지표-필드 매핑 완전성 확인
- **산출물**: ETL(추출·변환·적재) 소스-타겟 매핑 정의서

## 3. 실무 설계 시 주의할 점

### 3-1. 그레인을 절대 섞지 마세요
- 하나의 팩트 테이블에 서로 다른 그레인(예: 일별 데이터 + 월별 데이터)이 공존하면, 집계 시 **이중 계산(Double Counting) 등 잘못된 결과**가 발생합니다.
- 그레인이 다르면 **별도의 팩트 테이블**로 분리합니다.

### 3-2. 팩트 테이블 유형과 측정값 가산성을 구분하세요
서로 다른 축의 개념이므로 혼동하지 않습니다.

**(A) 팩트 테이블 유형** — 행이 표현하는 시간 구조
| 유형 | 설명 |
|---|---|
| 트랜잭션 (Transaction) | 개별 사건 1건 = 1행 |
| 주기적 스냅샷 (Periodic Snapshot) | 정해진 주기(일/월 등) 시점 상태 |
| 누적 스냅샷 (Accumulating Snapshot) | 프로세스 생애주기(다단계 마일스톤)를 한 행에 누적 |

**(B) 측정값 가산성(Additivity)** — 측정값을 차원 기준으로 합산할 수 있는가
| 구분 | 설명 | 예 |
|---|---|---|
| 가산 (Additive) | 모든 차원에서 SUM 가능 | 매출 금액, 판매 수량 |
| 준가산 (Semi-additive) | 일부 차원(특히 **시간축**)에서는 SUM 불가 | 재고 잔액, 계좌 잔고 |
| 비가산 (Non-additive) | SUM 자체가 무의미 | 비율, 평균, 단가 |

> 준가산·비가산 측정값은 재집계 시 오류를 유발하므로, **비율/평균/누계는 원자 값으로 저장하고 조회 시 재계산**하거나(뷰·시맨틱 레이어), 물리 저장이 필요하면 **분자·분모를 함께 저장**해 상위 집계에서도 정확성을 유지하도록 설계합니다.

### 3-3. 그레인 확정 전에 차원을 먼저 고정하지 마세요
- 2단계(그레인)를 확정하기 전에 3단계(차원)를 먼저 골라 고정하면, 나중에 데이터 레벨이 맞지 않아 모델을 재설계해야 합니다.
- 요구사항 수집 단계에서 후보 차원을 파악하는 것은 무방하나, **최종 확정은 반드시 그레인 이후**입니다.

### 3-4. 공유 차원(Conformed Dimension)을 우선 정의하세요
- 여러 비즈니스 프로세스(팩트)에서 동일하게 재사용되는 차원(예: 날짜, 고객, 조직)은 **하나의 표준 정의로 공유**합니다.
- 이것이 킴볼 버스 아키텍처의 핵심이며, 팩트 간 교차 분석(drill-across)을 가능하게 합니다.

# 모던 엔터프라이즈 데이터 아키텍처 설계 가이드

> **문서 읽기 순서: 1단계 → 2단계 → 3단계**

> - **2단계 (현재 문서)**: 클라우드 DWH / Modern Data Stack 일반 설계 원칙(벤더 중립 구현 패턴).
> - **3단계**: [`굿네이버스_snowflake 기반 엔터프라이즈 데이터 아키텍처 설계 가이드.md`](굿네이버스_snowflake%20기반%20엔터프라이즈%20데이터%20아키텍처%20설계%20가이드.md) — 본 원칙의 Snowflake(GN_DW) 구현 표준.

본 문서는 클라우드 데이터 웨어하우스(CDW) 및 모던 데이터 스택(Modern Data Stack) 환경에서 현업의 셀프서비스 BI와 생성형 AI(LLM) 활용을 동시에 지원하기 위한 데이터 플랫폼 설계 표준을 정의합니다.

## 1. 아키텍처 핵심 원칙

* **요구사항에 맞는 수집 파이프라인 최적화:** 초고속 실시간 처리가 필수적인 워크로드가 아니라면, 불필요한 스트리밍 인프라 도입(오버엔지니어링)을 지양하고 **배치(Batch) 및 마이크로 배치 중심의 ELT 파이프라인**으로 구성하여 인프라 비용과 운영 복잡도를 낮춥니다.
* **소비 계층(Consumption Layer)의 다원화:** 데이터를 소비하는 주체가 '사람(대시보드)'인지 'AI(LLM)'인지에 따라 제공하는 데이터의 형태와 논리적 구조를 엄격히 분리합니다.
* **데이터 관측성(Data Observability) 내재화:** 대부분의 모던 클라우드 DB는 분산 처리 성능을 위해 물리적인 제약 조건(PK/FK)을 강제하지 않습니다. 따라서 파이프라인 내부에 자체적인 데이터 품질 검증 단계를 반드시 포함해야 합니다.

---

## 2. 데이터 모델링 및 소비 계층 분리

분석 계층(Medallion 아키텍처의 Gold Layer)은 목적에 맞춰 두 가지 형태로 모델링하여 제공합니다.

### 2.1. 현업/BI 조회용: OBT (One-Big-Table) / 통합 플랫 뷰

데이터베이스의 정규화된 스타 스키마 구조를 현업 분석가나 마케터가 직접 다루기에는 조인(JOIN)의 복잡성이 높습니다. 이를 해결하기 위해 데이터를 넓게 펼친 비정규화 모델을 제공합니다.

* **목적:** 현업 사용자가 SQL 조인 없이 직관적으로 데이터를 탐색하고 BI 대시보드를 구성할 수 있도록 데이터 접근성을 극대화합니다.
* **구현 방식:**
* 초기에는 스토리지 비용과 갱신 부하를 막기 위해 논리적 뷰(Logical View)로 배포합니다.
* 조회 성능 병목이 발생하거나 쿼리 비용이 갱신 비용을 역전하는 시점에 데이터 변환 도구(예: dbt, Airflow 기반의 배치)를 활용하여 **물리적 테이블(Materialized Table)로 구체화**합니다.



### 2.2. AI 및 LLM(NL-to-SQL)용: 시맨틱 레이어 (Semantic Layer)

LLM에게 OBT 형태의 원천 데이터를 직접 읽게 하는 것은 토큰 비용을 낭비하고 환각(Hallucination)을 유발하는 안티 패턴입니다. LLM 기반의 데이터 에이전트는 데이터 자체가 아닌 **메타데이터(스키마, 지표 정의)를 기반으로 SQL을 생성**해야 합니다.

* **목적:** LLM이 비즈니스 용어와 데이터 구조를 정확히 매핑하여 오차 없는 집계 쿼리를 생성하도록 유도합니다.
* **구현 방식:**
* 평탄화된 뷰가 아닌 **코어 스타 스키마(Fact & Dimension)** 위에 시맨틱 레이어를 구축합니다.
* 조인 관계, 데이터의 최소 단위(Grain), 비가산성 지표(예: 이탈률, 평균 단가), 파생 지표(Derived Metric)의 계산 로직을 시맨틱 레이어에 명시적으로 코딩하여 중앙 통제합니다.
* **비용 및 성능:** LLM 연산에 소모되는 토큰은 스캔하는 데이터의 행(Row) 수가 아니라 시맨틱 계층에 정의된 메타데이터의 복잡도에 비례하므로, 명확하고 간결한 지표 정의가 필수적입니다.



| 구분 | 소비 주체 | 제공 형태 | 목적 및 특징 |
| --- | --- | --- | --- |
| **현업/BI용** | 사람, 대시보드 | OBT (통합 플랫 뷰) | 복잡한 JOIN 생략, 직관적인 탐색 및 필터링 제공 |
| **AI/LLM용** | LLM 기반 에이전트 | 시맨틱 레이어 (Semantic Layer) | 스타 스키마 기반 메타데이터 제공, 정확한 쿼리(SQL) 생성 유도 |

## 3. 리스크 방어 및 데이터 정합성 (Data Quality & Governance)

모던 CDW 환경에서 가장 경계해야 할 위험은 에러 로그 없이 지표가 부풀려지는 Fan-out(데이터 증식) 현상입니다.

* **Fan-out 방지 및 논리적 무결성 검증:**
* 클라우드 기반 데이터 플랫폼은 구조상 Primary Key(PK)와 Foreign Key(FK)를 '선언'만 할 뿐 시스템 레벨에서 중복을 차단(Enforce)하지 않는 경우가 많습니다.
* 차원(Dimension) 테이블의 키가 중복될 경우, 조인 과정에서 측정값(Fact)이 배수로 뻥튀기되는 치명적인 오류가 발생합니다.
* **대응:** 데이터 파이프라인의 적재 완료 직후, 데이터 품질 모니터링 도구(Data Quality Testing)나 검증 쿼리를 돌려 유일성(Uniqueness)과 참조 무결성을 강제 검사하는 로직을 추가합니다.


* **SCD (Slowly Changing Dimension) 이력 관리:**
* 변경 이력이 남는 차원(회원 등급, 조직, 상품 카테고리 등)은 윈도우 함수(Window Function) 등을 활용해 중복 제거(Dedup) 로직을 엄격하게 적용해야 합니다.
* **as-was vs 현재버전 기준을 명확히 결정**해야 합니다.
  - **현재버전(Current Snapshot)**: 팩트가 불변 업무키(Durable Key)만 보유 → 현재 속성만 제공 가능. 과거 시점 분석 불가.
  - **as-was(Historical)**: 팩트가 버전 대리키(Surrogate Key) 보유 → 사건 발생 당시의 속성 재현 가능.
  - 어느 쪽을 선택할지는 **팩트 설계 시 결정**해야 하며, 사후 변경은 스키마 수정을 수반합니다. 혼용 금지.


* **소비 계층 접근 통제 및 PII 보호 (RBAC & Data Masking):**
* 분석 계층(Gold)의 플랫 뷰·시맨틱 레이어에는 개인식별정보(PII) — 성별, 지역, 연령대, 회원번호 등 — 가 노출될 수 있습니다.
* **접근 롤 분리(RBAC)**: 현업 조회용 뷰와 원본 스타 스키마에 대한 접근 권한을 역할(Role) 단위로 분리합니다.
* **데이터 마스킹(Data Masking)**: 비인가 롤에는 PII 컬럼을 가림 처리하는 마스킹 정책을 파이프라인 설계 시 함께 정의합니다. 분석 편의보다 거버넌스 요건이 선행입니다.

## 4. 파이프라인 및 성능 최적화 (Pipeline & Performance)

* **선언적 데이터 변환 (Declarative ELT):**
* 데이터 웨어하우스 내부의 컴퓨팅 파워를 활용하여 Raw 데이터(Bronze) → 정제 데이터(Silver) → 분석용 모델(Gold)로 이어지는 변환 과정을 배치 파이프라인으로 자동화합니다.


* **파티셔닝 (Partitioning / Pruning):**
* 스캔하는 바이트 수와 컴퓨팅 자원을 줄이기 위한 핵심 방어 기제입니다. 파티셔닝/프루닝은 **논리 뷰가 아니라 물리 테이블**(대용량 팩트 테이블 또는 물리화된 OBT)에 적용됩니다. 논리 뷰는 조회 시 predicate를 하위 물리 테이블로 전달할 뿐이므로, 조회 필터가 가장 빈번한 기준(예: 발생일자, 주요 카테고리)으로 **하위 팩트 테이블**의 파티셔닝/정렬 전략을 수립합니다.


* **클러스터링 및 인덱싱 (Clustering):**
* 초기 설계부터 과도하게 정렬 키를 부여하지 않습니다. 시스템 런칭 후 실측 모니터링을 통해 조회 성능 병목이 확인된 테이블에 한정하여 선택적으로 적용(Post-optimization)합니다.

## 5. 설계 및 구축 우선순위 (Roadmap)

안정적인 데이터 플랫폼 구축을 위해 다음 순서로 프로젝트를 전개합니다.

1. **시맨틱 레이어 모델링 (최우선):** LLM 및 전사 지표 표준화의 핵심이므로, 코어 스타 스키마 위에 비즈니스 차원과 측정값(Metric)을 명확히 정의합니다.
2. **데이터 무결성 검증 파이프라인 구축:** PK/FK 미강제로 인한 통계 오류(Fan-out)를 방지하기 위한 정합성 테스트 로직을 세팅합니다.
3. **ELT 파이프라인 자동화:** 배치 스케줄러를 통해 Silver에서 Gold 레이어로의 데이터 변환 프로세스를 구축합니다.
4. **현업용 OBT(통합 플랫 뷰) 구성:** BI 사용자 접근성을 위한 비정규화 뷰를 논리적으로 정의합니다. (물리화는 보류)
5. **성능 튜닝 (Post-Tuning):** 운영 로그를 바탕으로 스토리지 파티셔닝 및 클러스터링을 사후 최적화합니다.

## 0. 3줄 요약

- GOLD는 별(star) 스키마입니다. 숫자를 담는 **팩트(FACT) 9개**와 분류축을 담는 **차원(DIM) 15개**, 총 **24개 테이블**로 구성됩니다.
- 테이블 사이의 관계(FK)는 `06_DDL.sql` 하단 `[관계 제약]` 섹션에서 **정보성(informational) FK 35개**로 선언해 두었습니다. Snowflake는 이 FK를 **실제로 강제하지 않습니다** — ERD를 그리고 BI 도구가 관계를 자동 인식하게 하는 문서·메타 용도입니다.
- 일부 관계는 일부러 FK를 **안 걸었습니다**(회원키·월키). 걸 수 없는 기술적 이유가 있으며, 대신 올바른 조인 방법을 이 문서와 DDL 주석에 적어 두었습니다.

## 1. 왜 FK를 걸어도 "강제"되지 않는가

Snowflake는 전통적 RDBMS와 다릅니다. `PRIMARY KEY`, `UNIQUE`, `FOREIGN KEY` 제약을 **선언할 수는 있지만, 실행 시 검사하지 않습니다.** 유일하게 강제되는 것은 `NOT NULL` 뿐입니다.

그렇다면 왜 FK를 선언했을까요?

1. **ERD 자동 생성** — 관계 정보가 메타데이터에 저장되어, 툴이 테이블 관계도를 그릴 수 있습니다.
2. **BI 도구 관계 인식** — Tableau·Power BI 등이 조인 관계를 자동으로 잡아냅니다.
3. **인수인계 문서화** — 어느 컬럼이 어느 차원을 가리키는지 코드 자체가 설명서가 됩니다.
4. (옵션) **쿼리 옵티마이저 최적화** — `RELY` 힌트를 주면 불필요한 조인을 생략해 속도가 빨라집니다. **단, 지금은 쓰지 않았습니다(아래 참조).**

## 2. 왜 `NORELY`(옵티마이저 신뢰 안 함)로 두었는가 — 중요

`RELY`를 붙이면 옵티마이저가 "이 FK는 무결성이 보장된다"고 **믿고** 조인을 생략합니다. 그런데 Snowflake는 FK를 강제하지 않으므로, **데이터가 실제로 어긋나 있으면 잘못된 결과(누락·오답)가 조용히 나옵니다.**

현재 GOLD는 아직 **데이터가 적재·검증되기 전**입니다. 이 단계에서 `RELY`를 켜는 것은 위험합니다. 그래서 모든 FK를 `NOT ENFORCED NORELY`(기본값, 명시적으로 표기)로 선언했습니다.

> **후임자 할 일**: SILVER→GOLD 적재 파이프라인이 참조 무결성을 실제로 보장하게 된 뒤(예: 적재 시 orphan 검증 통과), 그때 성능을 위해 FK를 `RELY`로 승격하는 것을 검토하세요. 검증 없이 켜지 마세요.

## 4. 일부러 FK를 안 건 관계 — 반드시 이해하고 넘어갈 것

Snowflake FK는 **가리키는 대상 컬럼이 PK 또는 UNIQUE여야** 선언할 수 있습니다. 아래 두 관계는 대상이 "비유일(여러 행이 같은 값)"이라 FK를 걸 수 없습니다. 이것은 실수가 아니라 **의도된 설계**이며, 원본 설계도(CSV 인벤토리)의 `※비강제` 표기와 정확히 일치합니다.

### 4-1. `MEMBER_DK` (회원 불변키) — FMM·FME·FSE·FEP·DIM_MEMBER_IDENTITY

- **왜 못 거나**: `DIM_MEMBER`는 SCD2(이력관리) 차원이라 한 회원이 상태 변경 이력만큼 **여러 행**으로 존재합니다. 따라서 `MEMBER_DK`는 그 테이블에서 유일하지 않습니다(PK는 `MEMBER_SK`).
- **왜 대리키(MEMBER_SK)로 FK 걸지 않았나**: 팩트는 재적재·시점정합 안정성을 위해 변하지 않는 업무 불변키 `MEMBER_DK`를 보관합니다. 대리키를 팩트에 심으면 SCD2 버전이 바뀔 때마다 팩트를 다시 매핑해야 합니다.
- **올바른 조인 방법**:
  - 현재 상태 기준: `DIM_MEMBER`에서 `IS_CURRENT = TRUE`인 행과 `MEMBER_DK`로 조인.
  - 사건 시점 기준: 팩트의 날짜가 `EFFECTIVE_FROM ~ EFFECTIVE_TO` 구간에 들어가는 회원 버전행과 조인.

### 4-2. `MONTH_KEY` (월 키, YYYYMM) — FMM·FTG_D·FTG_B·FBD

- **왜 못 거나**: 날짜 차원 `DIM_DATE`는 하루 1행(PK=`DATE_SK`, YYYYMMDD)입니다. `MONTH_KEY`는 한 달에 약 30개 행이 공유하므로 유일하지 않습니다.
- **올바른 조인 방법**: `DIM_DATE`에서 월초 행만 걸러 조인(예: `DAY = 1`)하거나, 월 단위 표준 뷰를 경유합니다.
- **향후 개선 여지**: 월 단위 전용 차원(`DIM_MONTH`)을 신설하면 FK를 걸 수 있습니다. 다만 "테이블 24개 고정" 원칙에 걸려 현재는 보류했습니다. (`03_테이블 설계.md`의 open 항목 참조)

### 4-3. 팩트 테이블의 PK/UNIQUE — 미설정

- 각 팩트의 논리적 grain(1행의 의미)은 확정 전이고, 적재는 ETL 멱등성(같은 데이터를 다시 넣어도 결과 동일)에 의존하도록 설계했습니다. 이 단계에서 PK를 잘못 선언하면 오히려 오해를 부릅니다.
- 대신 각 팩트의 grain은 `CREATE TABLE`의 `COMMENT`에 적어 두었습니다. grain이 확정되면 `UNIQUE (NORELY)`로 선언하는 것을 검토하세요.

## 7. 배포 시 주의 (실행 순서)

1. `USE DATABASE GN_DW; USE SCHEMA GOLD;`
2. DIM 15개 → FACT 9개 순서로 `CREATE TABLE` (DDL에 그 순서로 배열됨).
3. **테이블 24개가 모두 생성된 뒤** `[관계 제약]`의 `ALTER TABLE ... ADD ... FOREIGN KEY`를 실행. (부모 테이블이 먼저 있어야 FK가 붙습니다.)
4. 검증: 테이블 24개(`INFORMATION_SCHEMA.TABLES`), FK 35개(`SHOW IMPORTED KEYS` 또는 `INFORMATION_SCHEMA.TABLE_CONSTRAINTS` 카운트).

### ⚠️ 재실행 규칙 — 반드시 지킬 것
- **`06_DDL.sql`은 항상 위→아래로 "전체 일괄" 실행하세요.** `CREATE OR REPLACE`가 테이블을 다시 만들면서 기존 FK를 모두 제거하고, 이어지는 `ALTER`가 다시 부여하므로 전체 실행은 언제나 안전(멱등)합니다.
- **하지 말 것**:
  - FK 섹션만 따로 다시 실행 → "제약이 이미 존재" 오류가 납니다.
  - 특정 DIM만 `CREATE OR REPLACE` → 그 DIM을 참조하던 자식 팩트의 FK가 조용히 사라집니다.
- 일부만 고쳐야 한다면, 해당 FK를 `ALTER TABLE ... DROP CONSTRAINT <이름>`으로 지운 뒤 다시 `ADD`하세요.

### 타입 정합성 (검증 완료)
FK 35개 전부 **자식 컬럼 타입 = 부모 PK 타입**임을 확인했습니다. `DATE_SK`는 양쪽 모두 `NUMBER(8,0)`, 나머지 대리키(`_SK`)는 양쪽 모두 `NUMBER(38,0)`. 향후 타입을 바꿀 때는 **부모·자식을 함께** 변경해야 FK가 유효하게 유지됩니다.

## 5. ETL 오케스트레이션 (dbt pipeline)

> **라이브(2026-07-22): ETL은 dbt 프로젝트 `GN_DW.OPS.DW_PIPELINE`으로 운영.** 구설계의 Serverless Task DAG 4종(`TASK_VALIDATE_BRONZE`→`TASK_REFINEMENT_ROOT`→`TASK_LOAD_GOLD`→`TASK_FINALIZER`)·정제 프로시저·`ETL_LOG`는 **폐기**되었다(SHOW TASKS/PROCEDURES 실측 0).

```yaml
pipeline:
  engine: dbt
  project: GN_DW.OPS.DW_PIPELINE
  models: 65   # SILVER 32 → GOLD star 24 → WIDE VIEW 9
  flow: "BRONZE_<SOURCE> → SILVER 32 → GOLD star 24 → WIDE VIEW 9"
  execution: "dbt run / dbt build (dbt 프로젝트 실행). 워크스페이스 build green 확인."
  quality_gate: "dbt schema tests(not_null/unique/relationships). relationships=severity:warn(메달리온 BP), 핵심 PK=error"
run_principles:
  - "멱등성: dbt run이 table CREATE OR REPLACE / view 재생성 → 재실행 안전"
  - "센티넬 SK=0 라우팅으로 미매칭 차원키 흡수(파이프라인 중단 없이 관측)"
  - "스캐폴드 팩트(FACT_TARGET_BIZ)는 원천 미입고 시 0행 통과"
future_scheduling:
  - "정기 스케줄은 dbt 프로젝트를 감싸는 CREATE TASK(cron)로 래핑 예정 — 현재 수동/온디맨드 실행"
  - "Task 도입 시 Serverless(EXECUTE MANAGED TASK ON ACCOUNT, 01_환경_Role.md 2.2)"
  - "BRONZE 신선도 기반 트리거(Stream/WHEN)로 확장 가능"
deploy_ref: "10_dbt_pipeline/00_배포운영_통합_*.md · deploy_dbt_project.sql"
```

## 7. 보안 세팅 (security)

### 7.1 네트워크 (network)

```yaml
network_rules:   # GN_DW.SECURITY 스키마
  - { id: NR_OFFICE_IP, purpose: 사무실 IP 대역 허용, type: "IPV4 (INGRESS)" }
  - { id: NR_VPN_IP, purpose: VPN 접속 IP 허용, type: "IPV4 (INGRESS)" }
  - { id: NR_SERVICE_IP, purpose: 서비스/ETL 서버 IP, type: "IPV4 (INGRESS)" }
network_policies:
  - { id: NP_GN_DW_ACCOUNT, applies_to: Account, desc: "사무실+VPN+서비스 IP만 허용" }
  - { id: NP_GN_DW_SERVICE, applies_to: GN_DW_SERVICE 유저, desc: "서비스 IP만 허용(더 엄격)" }
apply_order:
  - "Network Rule 생성 -> Network Policy 생성 -> 테스트(본인 IP 포함 확인)"
  - "Account 활성화(ALTER ACCOUNT SET NETWORK_POLICY)"
warning: "본인 IP를 ALLOWED에 미포함 시 즉시 잠김. 반드시 테스트 후 적용 (R1)"
```

masking_layer:
  - "SILVER 물리 테이블 컬럼에 직접 적용 (View REPLACE 시 정책 단절 방지)"

## 8. 모니터링 세팅 (monitoring)

> **배포 상태(2026-07-22 실측):** 아래 Resource Monitor·Alert·Cost View는 **설계안(미배포)**이다. 현재 Warehouse 3종에 Resource Monitor 미연결(SHOW WAREHOUSES: resource_monitor=null). 운영 승격 시 아래 사양대로 생성한다.

### 8.1 Resource Monitor (resource_monitors)

> 계정 레벨 객체 (특정 스키마 미소속).

```yaml
resource_monitors:
  - { id: RM_ETL, target: GN_DW_ETL_WH, monthly_credit_limit: 200, triggers: "75% 알림, 90% SUSPEND, 100% SUSPEND_IMMEDIATE" }
  - { id: RM_ANALYTICS, target: GN_DW_ANALYTICS_WH, monthly_credit_limit: 500, triggers: "75% 알림, 90% 알림, 100% SUSPEND" }
  - { id: RM_DEV, target: GN_DW_DEV_WH, monthly_credit_limit: 100, triggers: "80% 알림, 100% SUSPEND" }
  - { id: RM_ACCOUNT, target: 전체 계정, monthly_credit_limit: 1000, triggers: "80% 알림, 95% SUSPEND" }
```

### 8.2 Alert (alerts)

> Alert는 운영 승격 시 `GN_DW.OPS`에 배치(설계안). ⚠️ 구설계의 `ETL_LOG` 기반 조건은 dbt 전환으로 폐기 — dbt run/test 결과(로그·artifacts) 기반 실패 알림으로 대체한다.

```yaml
alerts:   # 설계안 (미배포)
  - { id: ALERT_DBT_FAILURE, condition: "dbt run/test 실패(로그 기준)", action: 이메일 알림 }
  - { id: ALERT_LONG_QUERY, condition: "쿼리 실행 > 30분", action: 이메일 알림 }
  - { id: ALERT_BRONZE_STALE, condition: "BRONZE 최종 업데이트 > 24시간", action: 이메일 알림 }
```
